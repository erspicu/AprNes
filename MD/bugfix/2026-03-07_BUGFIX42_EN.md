## Fix: Sprite Size Latching at Dot 261 (P18, $0489, PASS)

### Problem
When `$2000` is written during HBlank to change `Spritesize8x16`, the sprite 0 CHR
tile data should be fetched using the sprite size that was in effect at the time of
the hardware's tile fetch (dots 261-264), not the value at dot 0 of the next scanline.

Our `PrecomputeSprite0Line()` always used the live `Spritesize8x16` value at dot 0,
which meant early and late HBlank writes to `$2000` produced identical results.
The test expects different behavior depending on whether `$2000` was written before
or after the sprite 0 CHR fetch window.

### Root Cause
On real NES hardware, sprite 0's CHR pattern is fetched at dots 261-264 of each
scanline's HBlank phase. The `Spritesize8x16` value at dot 261 determines:
1. Whether the sprite is treated as 8×8 or 8×16 for Y range checking
2. The tile address calculation (single tile vs tile pair)

If `$2000` is written BEFORE dot 261 → new size used for fetch
If `$2000` is written AFTER dot 261 → old size used for fetch

### Fix

**PPU.cs: ppu_rendering_tick** — Latch `Spritesize8x16` at dot 261:
```csharp
if (ppu_cycles_x == 261) spriteSizeLatchedForFetch = Spritesize8x16;
```

**PPU.cs: PrecomputeSprite0Line** — Use latched value instead of live value:
```csharp
bool sprSize16 = spriteSizeLatchedForFetch;
int height = sprSize16 ? 15 : 7;
// ... all subsequent Spritesize8x16 references replaced with sprSize16
```

### Test Details
The "Suddenly Resize Sprite" test has 5 sub-tests:
- Test 1: Prerequisite (sprite 0 hit verification)
- Test 2: 8px→16px, `$2000` written before dot 261 → hit expected
- Test 3: 8px→16px, `$2000` written after dot 261 → no hit expected (8px tile empty)
- Test 4: 16px→8px, `$2000` written before dot 261 → no hit expected (8px, out of range)
- Test 5: 16px→8px, `$2000` written after dot 261 → hit expected (16px tile used)

The latch correctly distinguishes tests 4 vs 5 (the failing pair in our emulator).

### Files Modified
- `PPU.cs`: Added `spriteSizeLatchedForFetch` field, latch at dot 261 in
  `ppu_rendering_tick()`, `PrecomputeSprite0Line()` uses latched value
