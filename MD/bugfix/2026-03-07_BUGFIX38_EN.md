# BUGFIX38: OAM DMA deferred execution (INC $4014 fix)

**Date**: 2026-03-07
**Affected test**: AccuracyCoin P18 $0480 (INC $4014) err=3 → PASS
**Baseline change**: AC 112/136 → 113/136 PASS, blargg 174/174 unchanged

## Problem Description

`INC $4014` (RMW instruction) produces two writes to $4014:
1. Dummy write: writes back original value → triggers OAM DMA (513+ cycles)
2. Real write: writes incremented value → triggers a second OAM DMA

On real NES hardware, DMA cannot start during a CPU write cycle, so only one DMA executes (using the page number from the second write).

## Fix Details

### PPU.cs
- `ppu_w_4014()` changed to only set `oamDmaPending = true` and `oamDmaPendingPage = value`
- Added `oamDmaExecute()` containing the original DMA execution logic

### MEM.cs
- `Mem_r()` and `ZP_r()` check `oamDmaPending` at the start; if pending, execute the DMA
- DMA only starts on the next read cycle, matching real hardware behavior

### APU.cs
- Added `oamDmaPending` and `oamDmaPendingPage` fields
- Initialized pending flag in `initAPU()`

## Test Results
- blargg 174/174 PASS (no regression)
- AccuracyCoin 113/136 PASS (+1: INC $4014)
