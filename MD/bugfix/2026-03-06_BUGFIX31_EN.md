**Date**: 2026-03-06

## Fix Details

### Load DMA countdown changed to parity-dependent

**File**: `AprNes/NesCore/APU.cs`

**Problem**: The scheduling delay for Load DMA (triggered by $4015 write) was fixed at 3 APU cycles,
but Mesen2 uses `CycleCount & 1` to determine whether the delay is 2 or 3:

```cpp
// Mesen2: DeltaModulationChannel.cpp SetEnabled()
if((_console->GetCpu()->GetCycleCount() & 0x01) == 0) {
    _transferStartDelay = 2;
} else {
    _transferStartDelay = 3;
}
```

**Fix**:
```csharp
// Old: dmcLoadDmaCountdown = 3;
// New:
dmcLoadDmaCountdown = (apucycle & 1) != 0 ? 2 : 3;
```

Mapping: `apucycle` odd = Mesen2 CycleCount even → delay 2;
`apucycle` even = Mesen2 CycleCount odd → delay 3.

## AccuracyCoin Test 82 Investigation Conclusion

Also investigated the hang in AccuracyCoin test 82 (IFlagLatency).

**Root Cause**: The DMC DMA stolen cycle calculation used `cpuBusIsWrite` as a proxy for GET/PUT phase.
In AccuracyCoin's clockslide sequence, 9 out of 79 DMAs fell on JSR push (write cycles),
each miscounting by 1 stolen cycle, accumulating an 8-cycle drift that prevented DMA from aligning to the $4013 opcode fetch.

**Conclusion**: The fix requires true GET/PUT phase tracking (replacing `cpuBusIsWrite` with `apucycle` parity),
but both parity directions currently break the sprdma tests. The root cause is that `apucycle`'s initial offset
does not match the actual hardware phase. This requires further calibration of the initial offset and is left as future work.
