# BUGFIX45: $2002 Flag Clear Timing Stagger (M2 Duty Cycle Effect)

**Date**: 2026-03-07
**AccuracyCoin**: 117 → 118/136 (+1)
**blargg**: 174/174 (no regression)

---

## Fix: Sprite flags clear 1 dot before VBL flag

### Problem
AccuracyCoin P18 "$2002 flag clear timing" test was failing (err=1).
The test checks that sprite flags (sprite 0 hit, sprite overflow) appear to clear
slightly before the VBL flag on the pre-render line.

### Root Cause
On real hardware, $2002 reads sample VBL flag at M2 rise and sprite flags at M2 fall.
The RP2A03G M2 duty cycle is 15/24, meaning sprite flags are read ~1.875 PPU dots after
VBL flag. This creates an observable timing difference: sprite flags appear to clear
~2 PPU dots earlier than VBL.

Our emulator cleared all three flags simultaneously at scanline 261, dot 2.

### Fix
Split the flag clear into two phases:
- **Dot 1**: Clear `isSprite0hit` and `isSpriteOverflow`
- **Dot 2**: Clear `isVblank` (unchanged)

This matches the M2 duty cycle effect without changing VBL timing.

### Previous Failed Attempts
- Moving VBL clear from dot 2 to dot 3 → regressed P17 VBlank End (-2)
- Moving all clears to dot 1 → no improvement (sprite flags need to be earlier than VBL, not same time)

### Why This Doesn't Regress P17
P17 "VBlank End" only checks the VBL flag (bit 7 of $2002). It never checks sprite flags.
Since VBL clear remains at dot 2, P17 is completely unaffected.

### Files Modified
- `PPU.cs`: Split flag clear at scanline 261 into dot 1 (sprite flags) and dot 2 (VBL)

### Also Fixed
- `run_ac_test.sh`: Fixed Left navigation (AccuracyCoin doesn't support Left wrapping). Now always uses Right navigation.
