# BUGFIX39: Controller Strobing parity (deferred $4016 write)

**日期**: 2026-03-07
**影響測試**: AccuracyCoin P14 $045F (Controller Strobing) err=4 → PASS
**基線變化**: AC 113/136 → 114/136 PASS, blargg 174/174 不變

## 問題描述

DEC $4016 (RMW instruction) 產生兩次連續寫入 $4016：
1. Cycle 5: 寫回原始值 (bit 0 = 1)
2. Cycle 6: 寫入遞減值 (bit 0 = 0)

真實 NES 硬體中，$4016 的 OUT pins 只在特定 CPU cycle parity (PUT cycle) 更新。
當 1-cycle strobe 對齊到 GET cycle 時，controller 不會被 strobe。

AccuracyCoin Test 3: OAM DMA 同步後立即 DEC $4016 → strobe 對齊 PUT cycle → 應觸發
AccuracyCoin Test 4: OAM DMA 同步 + LDA $00 (3 cycles) + DEC $4016 → strobe 對齊 GET cycle → 不應觸發

## 修復內容

### JoyPad.cs
- `gamepad_w_4016()` 改為 deferred 模型：設定 `strobeWriteValue` 和 `strobeWritePending`
- `strobeWritePending = (cpuCycleCount & 1) == 0 ? 1 : 2` (基於 CPU cycle count parity)
- 新增 `processStrobeWrite()` — 每個 CPU cycle 遞減 pending counter，歸零時執行實際 strobe 邏輯

### MEM.cs
- `tick()` 和 `tick_pre()` 中呼叫 `processStrobeWrite()`
- 確保每個 CPU cycle 都處理 pending writes

### CPU.cs
- `ResetInterrupt()` 中重置 `strobeWritePending` 和 `P1_LastWrite`

## 機制說明

參考 Mesen2 `NesControlManager::WriteRam()`：
```cpp
_writePending = (_console->GetMasterClock() & 0x01) ? 1 : 2;
```
`ProcessWrites()` 每個 CPU cycle 遞減 pending counter。

DEC $4016 的兩次寫入間隔只有 1 cycle：
- Parity A (pending=1): 第一次寫入在下一 cycle 處理後，第二次覆蓋 → latch 看到 1→0 → strobe
- Parity B (pending=2): 第一次寫入尚未處理，第二次覆蓋 → latch 只看到 0→0 → 不 strobe
