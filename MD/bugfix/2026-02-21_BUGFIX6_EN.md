## 27. Unofficial abs,X RMW Opcodes Missing Wrong-Page Dummy Read

**Commit**: `0fdd934`

### Symptom

blargg `instr_misc/04-dummy_reads_apu.nes` test fails, reporting incorrect dummy read behavior for 21 opcodes. The test uses the APU $4015 IRQ flag to detect whether the CPU read an I/O register at the correct cycle.

### Root Cause

The 6502 CPU, in Read-Modify-Write (RMW) instructions with absolute,X addressing, **regardless of page crossing**, must first perform a dummy read from the "wrong page" address (high byte uncorrected). This is cycle 4 of the 7 CPU cycles:

```
Cycle 1: fetch opcode
Cycle 2: fetch addr low (BAL)
Cycle 3: fetch addr high (BAH)
Cycle 4: read from BAH:BAL+X (wrong page, no carry fix) ← missing
Cycle 5: read from correct address
Cycle 6: write original value (dummy write)
Cycle 7: write modified value
```

The original implementation computed `(base + X)` directly as the final address, skipping the cycle 4 dummy read.

### Affected Opcodes

| Opcode | Instruction | Description |
|--------|-------------|-------------|
| 0x1F | SLO abs,X | ASL M then ORA |
| 0x3F | RLA abs,X | ROL M then AND |
| 0x5F | SRE abs,X | LSR M then EOR |
| 0x7F | RRA abs,X | ROR M then ADC |
| 0xDF | DCP abs,X | DEC M then CMP |
| 0xFF | ISC abs,X | INC M then SBC |

### Fix

Split the address calculation; before reading the correct address, perform a dummy read from the wrong-page address:

```csharp
// Before
ushort1 = (ushort)((Mem_r(r_PC++) | (Mem_r(r_PC++) << 8)) + r_X);
byte1 = Mem_r(ushort1);

// After
ushort2 = (ushort)(Mem_r(r_PC++) | (Mem_r(r_PC++) << 8));  // base address
ushort1 = (ushort)(ushort2 + r_X);                          // final address
Mem_r((ushort)((ushort2 & 0xFF00) | (ushort1 & 0x00FF)));   // dummy read: BAH:BAL+X (no carry fix)
byte1 = Mem_r(ushort1);                                      // real read
```

The wrong-page address is computed as: take the high byte of base, take the low byte of final,
i.e. `(BAH << 8) | ((BAL + X) & 0xFF)`. CPU hardware always performs this read regardless of page crossing.

## 28. Unofficial abs,Y RMW Opcodes Missing Wrong-Page Dummy Read

**Commit**: `0fdd934`

### Symptom

Same as Bug 27; `04-dummy_reads_apu.nes` reports incorrect dummy reads for these opcodes.

### Root Cause

Same issue as Bug 27, but using the Y register as the index. abs,Y RMW has the exact same cycle structure as abs,X; cycle 4 also requires a wrong-page dummy read.

### Affected Opcodes

| Opcode | Instruction | Description |
|--------|-------------|-------------|
| 0x1B | SLO abs,Y | ASL M then ORA |
| 0x3B | RLA abs,Y | ROL M then AND |
| 0x5B | SRE abs,Y | LSR M then EOR |
| 0x7B | RRA abs,Y | ROR M then ADC |
| 0xDB | DCP abs,Y | DEC M then CMP |
| 0xFB | ISC abs,Y | INC M then SBC |

### Fix

Same pattern as Bug 27, using `r_Y`:

```csharp
ushort2 = (ushort)(Mem_r(r_PC++) | (Mem_r(r_PC++) << 8));
ushort1 = (ushort)(ushort2 + r_Y);
Mem_r((ushort)((ushort2 & 0xFF00) | (ushort1 & 0x00FF))); // wrong-page dummy read
byte1 = Mem_r(ushort1);
```

## 29. Unofficial (ind),Y RMW Opcodes Missing Wrong-Page Dummy Read

**Commit**: `0fdd934`

### Symptom

Same as Bug 27; `04-dummy_reads_apu.nes` reports incorrect dummy reads for these opcodes.

### Root Cause

(indirect),Y addressing RMW instructions use 8 CPU cycles:

```
Cycle 1: fetch opcode
Cycle 2: fetch pointer address (ZP)
Cycle 3: fetch addr low from ZP
Cycle 4: fetch addr high from ZP+1
Cycle 5: read from BAH:BAL+Y (wrong page, no carry fix) ← missing
Cycle 6: read from correct address
Cycle 7: write original value (dummy write)
Cycle 8: write modified value
```

The original implementation added the Y offset directly to the indirect-fetched base to get the final address, skipping cycle 5. Additionally, the original code did not distinguish between the indirect address fetch and the final address computation:

```csharp
ushort1 = (ushort)((NES_MEM[byte2++] | (NES_MEM[byte2] << 8)) + r_Y);
```

### Affected Opcodes

| Opcode | Instruction | Description |
|--------|-------------|-------------|
| 0x13 | SLO (ind),Y | ASL M then ORA |
| 0x33 | RLA (ind),Y | ROL M then AND |
| 0x53 | SRE (ind),Y | LSR M then EOR |
| 0x73 | RRA (ind),Y | ROR M then ADC |
| 0xD3 | DCP (ind),Y | DEC M then CMP |
| 0xF3 | ISC (ind),Y | INC M then SBC |

### Fix

Split the indirect address fetch and Y offset computation:

```csharp
// Before
byte2 = Mem_r(r_PC++);
ushort1 = (ushort)((NES_MEM[byte2++] | (NES_MEM[byte2] << 8)) + r_Y);
byte1 = Mem_r((ushort)(ushort1));

// After
byte2 = Mem_r(r_PC++);
ushort2 = (ushort)(NES_MEM[byte2++] | (NES_MEM[byte2] << 8));  // indirect base
ushort1 = (ushort)(ushort2 + r_Y);                              // final address
Mem_r((ushort)((ushort2 & 0xFF00) | (ushort1 & 0x00FF)));       // wrong-page dummy read
byte1 = Mem_r(ushort1);                                         // real read
```

## 30. SHY/SHX/LAX Missing Dummy Read

**Commit**: `0fdd934`

### Symptom

Same as Bug 27; `04-dummy_reads_apu.nes` reports incorrect dummy reads for SHY, SHX, and LAX.

### Root Cause

Three different types of missing dummy reads:

**SHY (0x9C) / SHX (0x9E) — Write abs,X/Y**

SHY and SHX are write instructions using 5 CPU cycles.
Cycle 4 **regardless of page crossing** must read the wrong-page address (same as RMW), but the original implementation lacked this read.

```
Cycle 1: fetch opcode
Cycle 2: fetch addr low
Cycle 3: fetch addr high
Cycle 4: read from wrong-page address ← missing
Cycle 5: write to correct/glitched address
```

**LAX abs,Y (0xBF) / LAX (ind),Y (0xB3) — Read abs,Y**

LAX is a read instruction using 4+1 CPU cycles. A wrong-page dummy read is required **only on page crossing** (same as official LDA abs,Y), but the original implementation omitted this read.

### Fix

**SHY/SHX**: after computing the wrong-page address, before the page-cross correction, add a dummy read:

```csharp
// SHY (0x9C) fix
ushort1 = (ushort)(Mem_r(r_PC++) | (Mem_r(r_PC++) << 8));
byte1 = (byte)(r_Y & (((ushort1 & 0xff00) >> 8) + 1));
ushort1 = (ushort)((ushort1 & 0xff00) | (byte)(ushort1 + r_X));
Mem_r(ushort1);  // ← added dummy read at wrong-page address
if ((ushort1 & 0xff) < r_X) ushort1 = (ushort)((ushort1 & 0xff) | (byte1 << 8));
Mem_w(ushort1, byte1);
```

**LAX**: add page-cross conditional dummy read:

```csharp
// LAX abs,Y (0xBF) fix
ushort2 = (ushort)(Mem_r(r_PC++) | (Mem_r(r_PC++) << 8));
ushort1 = (ushort)(ushort2 + r_Y);
if ((ushort1 & 0xFF00) != (ushort2 & 0xFF00))  // page cross
    Mem_r((ushort)((ushort2 & 0xFF00) | (ushort1 & 0x00FF)));  // dummy read
r_X = r_A = Mem_r(ushort1);
```

## 31. Unofficial Opcodes TAS/SHA/LAS Not Implemented

**Commit**: `0fdd934`

### Symptom

When `instr_misc/04-dummy_reads_apu.nes` reaches opcode 0x9B, the emulator throws "unknown opcode 0x9B" and crashes.

### Root Cause

The following 4 unofficial opcodes are completely unimplemented and classified as unknown opcodes:

| Opcode | Mnemonic | Addressing | Description |
|--------|----------|------------|-------------|
| 0x9B | TAS/SHS | abs,Y | SP = A & X; write SP & (H+1) |
| 0x9F | SHA/AXA | abs,Y | write A & X & (H+1) |
| 0x93 | SHA/AXA | (ind),Y | write A & X & (H+1) |
| 0xBB | LAS/LAR | abs,Y | A = X = SP = M & SP |

These opcodes are unstable (behavior may vary across different NES hardware), but blargg's dummy read test exercises every opcode, so a basic implementation is required to avoid crashes.

### Fix

**TAS/SHS (0x9B)**: set SP = A & X, then write SP & (H+1) to the target address. Write abs,Y type; must include wrong-page dummy read:

```csharp
case 0x9B://TAS/SHS abs,Y
    ushort1 = (ushort)(Mem_r(r_PC++) | (Mem_r(r_PC++) << 8));
    r_SP = (byte)(r_A & r_X);
    byte1 = (byte)(r_SP & (((ushort1 & 0xff00) >> 8) + 1));
    ushort1 = (ushort)((ushort1 & 0xff00) | (byte)(ushort1 + r_Y));
    Mem_r(ushort1); // dummy read
    if ((ushort1 & 0xff) < r_Y)
        ushort1 = (ushort)((ushort1 & 0xff) | (byte1 << 8)); // page-cross glitch
    Mem_w(ushort1, byte1);
    break;
```

**SHA/AXA abs,Y (0x9F)**: write A & X & (H+1), write abs,Y type:

```csharp
case 0x9F://SHA/AXA abs,Y
    ushort1 = (ushort)(Mem_r(r_PC++) | (Mem_r(r_PC++) << 8));
    byte1 = (byte)(r_A & r_X & (((ushort1 & 0xff00) >> 8) + 1));
    ushort1 = (ushort)((ushort1 & 0xff00) | (byte)(ushort1 + r_Y));
    Mem_r(ushort1); // dummy read
    if ((ushort1 & 0xff) < r_Y)
        ushort1 = (ushort)((ushort1 & 0xff) | (byte1 << 8));
    Mem_w(ushort1, byte1);
    break;
```

**SHA/AXA (ind),Y (0x93)**: same as SHA but using (indirect),Y addressing:

```csharp
case 0x93://SHA/AXA (ind),Y
    byte2 = Mem_r(r_PC++);
    ushort2 = (ushort)(NES_MEM[byte2++] | (NES_MEM[byte2] << 8));
    ushort1 = (ushort)(ushort2 + r_Y);
    byte1 = (byte)(r_A & r_X & (((ushort2 >> 8) & 0xFF) + 1));
    Mem_r((ushort)((ushort2 & 0xFF00) | (ushort1 & 0x00FF))); // wrong-page dummy read
    if ((ushort1 & 0xFF00) != (ushort2 & 0xFF00))
        ushort1 = (ushort)((ushort1 & 0xFF) | (byte1 << 8));  // page-cross glitch
    Mem_w(ushort1, byte1);
    break;
```

**LAS/LAR (0xBB)**: read abs,Y type, A = X = SP = M & SP:

```csharp
case 0xBB://LAS/LAR abs,Y
    ushort2 = (ushort)(Mem_r(r_PC++) | (Mem_r(r_PC++) << 8));
    ushort1 = (ushort)(ushort2 + r_Y);
    if ((ushort1 & 0xFF00) != (ushort2 & 0xFF00)) // page cross
        Mem_r((ushort)((ushort2 & 0xFF00) | (ushort1 & 0x00FF))); // dummy read
    r_A = r_X = r_SP = (byte)(Mem_r(ushort1) & r_SP);
    if (r_A == 0) flagZ = 1; else flagZ = 0;
    if ((r_A & 0x80) != 0) flagN = 1; else flagN = 0;
    break;
```

### About the Page-Cross Glitch

Unstable write instructions like TAS/SHA/SHX/SHY have a special page-cross behavior:
when base + index crosses a page boundary, the written high byte is not H+1 (normal carry)
but is ANDed with the write value itself. This means on a page cross, the high byte of the
target address is overwritten with `value & (H+1)`, typically causing the write to land at a lower memory location.

```
Normal:   base=$10FF, Y=$01 → addr=$1100 (carry to H)
Glitched: base=$10FF, Y=$01, value=$05 → addr=$0500 (H = value & (H+1) = $05 & $11 = $01... actual depends on bus contention)
```
