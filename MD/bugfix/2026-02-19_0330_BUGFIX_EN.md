## 1. MMC3 IRQ Incorrect Clocking Condition

### Symptom
After entering a level in Super Mario Bros. 3, the background graphics display horizontal misalignment, corruption, or split-line failures.

### Root Cause
In `PPU.cs`, the trigger condition for the MMC3 IRQ timer was too restrictive: it only clocked when both "sprite pattern table at $1000" **and** "background pattern table at $0000" were satisfied simultaneously. This covers only one of the two pattern table configurations, so the IRQ would never fire under other configurations.

On real NES hardware, the MMC3 IRQ is triggered by a rising edge on PPU address bus signal A12 (low→high). This happens exactly once per visible scanline regardless of the pattern table configuration.

### Affected Location
`PPU.cs`: IRQ clocking check inside `ppu_step()`.

### Fix
Remove the superfluous pattern table condition. Clock once at cycle 260 of each scanline as long as rendering is enabled (`ShowBackGround || ShowSprites`) and the mapper is 4:

```csharp
// Before (condition too strict)
if (SpPatternTableAddr == 0x1000 && BgPatternTableAddr == 0 && mapper == 4)
    (MapperObj as Mapper004).Mapper04step_IRQ();

// After
if ((ShowBackGround || ShowSprites) && mapper == 4)
    (MapperObj as Mapper004).Mapper04step_IRQ();
```

## 2. MMC3 $C001 IRQ Reload Write Logic Error

### Symptom
Same as above (MMC3 game split-line position incorrect).

### Root Cause
In `Mapper004.cs`, a CPU write to `$C001` (odd address) should set the "reload counter on next A12 rising edge" flag (`IRQReset = true`). However, the original implementation incorrectly OR-ed the counter directly (`IRQCounter |= 0x80`), which has no meaningful effect.

NES hardware specification: a write to `$C001` causes the counter to reload from the latch value on the **next** A12 rising edge, not an immediate counter modification.

### Affected Location
`Mapper004.cs`: odd-address branch of `address < 0xe000` inside `MapperW_PRG()`.

### Fix
```csharp
// Before (wrong: directly modifies counter)
else if (address < 0xe000)
{
    IRQCounter |= 0x80;
    // ...
}

// After (correct: sets reload flag)
else if (address < 0xe000)
{
    IRQReset = true;
}
```

## 3. MMC3 $E000 IRQ Disable Write Logic Error

### Symptom
Same as above (MMC3 game IRQ rhythm disordered).

### Root Cause
In `Mapper004.cs`, a CPU write to `$E000` (even address) should disable IRQ (`IRQ_enable = false`). The original implementation additionally performed `IRQCounter = IRQlatchVal` (reload counter), which is an incorrect side effect.

NES hardware specification: the only effect of writing `$E000` is to clear the IRQ enable flag; the counter state must not be changed.

### Affected Location
`Mapper004.cs`: even-address branch of `address >= 0xe000` inside `MapperW_PRG()`.

### Fix
```csharp
// Before (extra counter reload)
else
{
    IRQ_enable = false;
    IRQCounter = IRQlatchVal; // ← wrong
}

// After
else
{
    IRQ_enable = false;
}
```

## 4. 8x16 Sprite Vertical Flip Tile Swap Missing

### Symptom
In MMC3 games, 8x16 sprites in vertical-flip mode have their top and bottom tile patterns reversed (the tile positions are not swapped along with the flip).

### Root Cause
An 8x16 sprite consists of two 8x8 tiles (top tile and bottom tile). During vertical flip, in addition to reversing the pixel rows within each tile, the two tiles themselves must also be swapped (top tile appears at the bottom, bottom tile appears at the top). The original code only reversed the row order but did not swap the tile indices.

### Affected Location
`PPU.cs`: 8x16 vertical-flip check inside `RenderSpritesLine()`.

### Fix
```csharp
if ((sprite_attr & 0x80) != 0)
{
    line_t = 7 - line;
    if (Spritesize8x16) tile_th_t ^= 1; // ← added: swap top/bottom tile (toggle bit0)
}
```

## 5. $2001 Left 8-Pixel Mask Not Implemented

### Symptom
In some games the leftmost 8 pixels should show the background color (masked), but background tiles or sprite pixels are displayed instead, causing edge flickering.

### Root Cause
PPU register `$2001` bit 1 (background) and bit 2 (sprites) control whether the leftmost 8 pixels are rendered. Games typically clear these bits while scrolling to avoid partial tiles appearing at the left edge. The original code contained no logic to read these two bits at all.

### Affected Location
- `PPU.cs`: `ppu_w_2001()`, `RenderBackGroundLine()`, `RenderSpritesLine()`.

### Fix

Add two static fields:
```csharp
static bool ShowBgLeft8 = true, ShowSprLeft8 = true;
```

Read them in `ppu_w_2001()`:
```csharp
static void ppu_w_2001(byte value)
{
    ShowBgLeft8  = (value & 0x02) != 0;
    ShowSprLeft8 = (value & 0x04) != 0;
    ShowBackGround = (value & 0x08) != 0;
    ShowSprites    = (value & 0x10) != 0;
}
```

Background rendering: when `ShowBgLeft8 = false` and pixel X < 8, output the background color (`ppu_ram[0x3f00]`) and set `Buffer_BG_array` to 0 (transparent).

Sprite rendering: when `ShowSprLeft8 = false` and `screenX < 8`, skip that pixel.

## 6. 8-Sprite-per-Scanline Limit Not Implemented

### Symptom
Some sprites that would be hidden on real NES hardware due to exceeding the 8-sprite-per-scanline limit are rendered by the emulator, causing sprites that should disappear (such as enemies or mask sprites) to be visible.

### Root Cause
The NES hardware's Secondary OAM can hold at most 8 sprites per scanline. During OAM evaluation, sprites are scanned in order from sprite 0 to sprite 63; earlier sprites take priority, and sprites 9 and beyond are not rendered. The original code rendered all 64 sprites with no limit.

Game designers rely on this limit: by intentionally placing more than 8 sprites, they cause certain sprites to exceed the limit and not be rendered, achieving masking or hiding effects.

### Affected Location
`PPU.cs`: `RenderSpritesLine()`.

### Fix
Redesign rendering as two passes:

```csharp
static void RenderSpritesLine()
{
    // Pass 1: scan OAM in order, select first 8 visible sprites, set overflow flag starting at 9th
    int* sel = stackalloc int[8];
    int selCount = 0, spriteCount = 0;
    int height = Spritesize8x16 ? 15 : 7;

    for (int oam_th = 0; oam_th < 64; oam_th++)
    {
        int raw_y = spr_ram[oam_th << 2];
        if (scanline <= raw_y || scanline - raw_y > height + 1) continue;
        if (++spriteCount == 9) isSpriteOverflow = true;
        if (selCount < 8) sel[selCount++] = oam_th;
    }

    if (!ShowSprites) return;

    // Pass 2: render in reverse OAM order (lower-index sprites overwrite higher-index)
    for (int si = selCount - 1; si >= 0; si--)
    {
        // ... render sprite sel[si]
    }
}
```

`stackalloc int[8]` allocates on the stack to avoid GC pressure.

## 7. Sprite Scan Range vs. Render Range off-by-1

### Symptom
After implementing fix #6, sprite rendering showed corruption: a row of garbage appeared at the top of some sprites (reading tile row -1), and the bottom row of sprites was missing.

### Root Cause
The Y field in NES OAM stores "display top scanline − 1", i.e. `y_loc = raw_y + 1`.

Pass 1 scanning used a range based on `raw_y` of `[raw_y, raw_y+height]`, but Pass 2 rendering used `y_loc = raw_y + 1` to compute `line = scanline - y_loc`. The two baselines differ by 1, causing:

| Case | Result |
|------|--------|
| `scanline == raw_y` selected in Pass 1 | `line = -1` → reads CHR data row -1 (out-of-bounds, garbage) |
| `scanline == raw_y + height + 1` not selected | Bottom row of sprite never rendered |

### Affected Location
`PPU.cs`: Pass 1 scan condition inside `RenderSpritesLine()`.

### Fix
Change the scan condition to use the "display baseline", consistent with `y_loc` in Pass 2:

```csharp
// Before (baseline is raw_y)
if (raw_y > scanline || scanline - raw_y > height) continue;

// After (baseline is raw_y+1, consistent with y_loc)
// OAM Y = display_top - 1; visible range is [raw_y+1, raw_y+1+height]
if (scanline <= raw_y || scanline - raw_y > height + 1) continue;
```

After the fix:
- When `scanline == raw_y + 1`: `line = 0` (correct row 0)
- When `scanline == raw_y + height + 1`: `line = height` (correct last row)

## 8. Background Fine Y Uses scrol_y Instead of vram_addr

### Symptom
In MMC3 games (e.g. Super Mario Bros. 3), after entering a level, the portion of the screen below a background split line shows horizontal row shifts or tile pattern misalignment ("screen rows jump to wrong positions").

### Root Cause
The `GetAttr()` function, when reading the pattern table for background tiles, computes fine Y (the pixel-row offset within a tile) using the following formula:

```csharp
// Wrong
array_loc = (ppu_ram[tileAddr] << 4) + BgPatternTableAddr + ((scanline + scrol_y) & 7);
```

`scrol_y` is only updated when the CPU writes to `$2005` (second write). SMB3's split-scroll effect works by writing to `$2006` directly inside the MMC3 IRQ handler, which updates `vram_addr` without touching `scrol_y`.

Result: for the portion below the split line, `scrol_y` holds a stale value, fine Y is computed incorrectly, and each tile reads CHR data from the wrong pixel row → background pattern shifts up or down per tile.

The correct NES PPU fine Y is always bits 12–14 of `vram_addr`, which is correct regardless of whether scrolling is set via `$2005` or `$2006`.

### Affected Location
`PPU.cs`: `array_loc` calculation inside `GetAttr()`.

### Fix
```csharp
// Before (scrol_y not updated by $2006)
array_loc = (ppu_ram[tileAddr] << 4) + BgPatternTableAddr + ((scanline + scrol_y) & 7);

// After (read fine Y directly from vram_addr bits 12-14)
array_loc = (ppu_ram[tileAddr] << 4) + BgPatternTableAddr + ((vram_addr >> 12) & 7);
```

**Compatibility with existing games**: For games without split scroll, `(scanline + scrol_y) & 7` and `(vram_addr >> 12) & 7` are mathematically equivalent, so the fix does not affect existing game behavior.

**Side-fix**: With correct `Buffer_BG_array` values, the occlusion check for behind-background sprites (priority=1) is also restored, simultaneously fixing "sprite visible when it should be hidden."

## 9. MMC3 IRQ Not Clocked on Pre-render Scanline 261

### Symptom
The split-screen effect in MMC3 games shifts by one scanline per frame, or the split-line height differs from expected by 1 (slight but accumulates visibly over time).

### Root Cause
The MMC3 IRQ counter is triggered by PPU A12 rising edges. On NES hardware this fires on **all rendering scanlines**, including:

- Visible scanlines: 0–239 (240 times/frame)
- Pre-render scanline: 261 (1 time/frame, total 241 times/frame)

The original code's condition `if (scanline < 240)` only clocked for 0–239, missing scanline 261 once per frame. One missed clock per frame means the effective IRQ latch value is 1 more than the game expects, causing the split line to fire on the wrong scanline.

### Affected Location
`PPU.cs`: IRQ clocking block inside `ppu_step()` (originally only within `if (scanline < 240)`).

### Fix
Add a clock for scanline 261 in the `else if` chain after `if (scanline < 240)`:

```csharp
if (scanline < 240)
{
    ...
    else if (ppu_cycles_x == 260)
    {
        if ((ShowBackGround || ShowSprites) && mapper == 4)
            (MapperObj as Mapper004).Mapper04step_IRQ();
    }
}
else if (scanline == 261 && ppu_cycles_x == 260)
{
    // pre-render scanline must also clock (hardware generates A12 rising edge here too)
    if ((ShowBackGround || ShowSprites) && mapper == 4)
        (MapperObj as Mapper004).Mapper04step_IRQ();
}
```
