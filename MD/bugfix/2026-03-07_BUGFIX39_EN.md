# BUGFIX39: Controller Strobing parity (deferred $4016 write)

**Date**: 2026-03-07
**Affected test**: AccuracyCoin P14 $045F (Controller Strobing) err=4 → PASS
**Baseline change**: AC 113/136 → 114/136 PASS, blargg 174/174 unchanged

## Problem Description

DEC $4016 (RMW instruction) produces two consecutive writes to $4016:
1. Cycle 5: writes back original value (bit 0 = 1)
2. Cycle 6: writes decremented value (bit 0 = 0)

On real NES hardware, the OUT pins of $4016 only update on a specific CPU cycle parity (PUT cycle).
When the 1-cycle strobe aligns to a GET cycle, the controller is not strobed.

AccuracyCoin Test 3: immediately after OAM DMA sync, DEC $4016 → strobe aligns to PUT cycle → should trigger
AccuracyCoin Test 4: OAM DMA sync + LDA $00 (3 cycles) + DEC $4016 → strobe aligns to GET cycle → should not trigger

## Fix Details

### JoyPad.cs
- `gamepad_w_4016()` changed to deferred model: sets `strobeWriteValue` and `strobeWritePending`
- `strobeWritePending = (cpuCycleCount & 1) == 0 ? 1 : 2` (based on CPU cycle count parity)
- Added `processStrobeWrite()` — decrements pending counter every CPU cycle, executes actual strobe logic when it reaches zero

### MEM.cs
- `tick()` and `tick_pre()` call `processStrobeWrite()`
- Ensures pending writes are processed every CPU cycle

### CPU.cs
- `ResetInterrupt()` resets `strobeWritePending` and `P1_LastWrite`

## Mechanism

Reference: Mesen2 `NesControlManager::WriteRam()`:
```cpp
_writePending = (_console->GetMasterClock() & 0x01) ? 1 : 2;
```
`ProcessWrites()` decrements the pending counter every CPU cycle.

The two writes of DEC $4016 are only 1 cycle apart:
- Parity A (pending=1): first write is processed on the next cycle, then second write overwrites → latch sees 1→0 → strobe
- Parity B (pending=2): first write not yet processed, second write overwrites → latch only sees 0→0 → no strobe
