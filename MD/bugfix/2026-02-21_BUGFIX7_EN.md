## 32. MMC3 (Mapper 004) IRQ Directly Calls IRQInterrupt in tick-on-access Architecture, Causing Screen Corruption

### Symptom

Super Mario Bros 3 (Mapper 004/MMC3) has severe screen corruption. This issue appeared after transitioning to the CPU tick-on-access cycle-accurate architecture.

### Root Cause

Mapper004's `Mapper04step_IRQ()` directly calls `NesCore.IRQInterrupt()` at PPU A12 rising edge time (at cycle 260 inside `ppu_step_new()`).

In the old architecture, `IRQInterrupt()` only set register state (push stack, set PC) and compensated cycles after the fact with `Interrupt_cycle = 7`. Although synchronously calling it from within a PPU step was still incorrect, the side effects were minimal since no actual bus operations occurred.

In the new tick-on-access architecture, `IRQInterrupt()` was changed to 7 real `Mem_r`/`Mem_w` calls (each of which triggers `tick()`). Calling it from inside a PPU step causes two serious problems:

1. **CPU state corrupted mid-instruction**: a CPU instruction is executing `Mem_r(addr)` → `tick()` → `ppu_step_new()` → `Mapper04step_IRQ()` → `IRQInterrupt()` modifies `r_PC` and `r_SP`, but the original instruction is still executing, completely corrupting CPU state.

2. **7 PPU/APU ticks swallowed**: `tick()`'s `in_tick` anti-recursion guard blocks the 7 nested `tick()` calls from within `IRQInterrupt()`'s `Mem_r`/`Mem_w`, causing 7 CPU cycles of PPU/APU advancement to be completely lost.

### Call Chain Analysis

```
CPU instruction executing
  └→ Mem_r(addr)
       └→ tick()                    ← in_tick = true
            └→ ppu_step_new() ×3
                 └→ (cycle 260) Mapper04step_IRQ()
                      └→ IRQInterrupt()          ← directly modifies PC/SP!
                           └→ Mem_r(r_PC)        ← tick() blocked by in_tick
                           └→ Mem_w(stack)        ← tick() blocked by in_tick
                           └→ ... (7 times total)
            └→ apu_step()
       └→ mem_read_fun[addr]        ← but PC/SP already corrupted
```

### Fix

Change to correct hardware behavior: mapper only asserts the /IRQ line; CPU polls IRQ sources at instruction boundaries.

#### 1. CPU.cs — Add mapper IRQ flag

```csharp
static public bool statusmapperint = false; // mapper IRQ line asserted (MMC3 etc.)
```

Remove the now-unneeded `FlagI_public` (originally only used by mapper to check the I flag).

Clear on reset:
```csharp
statusmapperint = false;
```

#### 2. Mapper004.cs — Set flag instead of direct call

```csharp
// Before
if (IRQCounter == 0 && IRQ_enable && NesCore.FlagI_public == 0)
    NesCore.IRQInterrupt();

// After
if (IRQCounter == 0 && IRQ_enable)
    NesCore.statusmapperint = true; // assert /IRQ line, polled at instruction boundary
```

Also remove the `FlagI_public == 0` check: on real hardware, the mapper does not care about the CPU's I flag — it only asserts the /IRQ line. The CPU decides at the instruction boundary whether to service the IRQ (based on the I flag).

On $E000 even write, acknowledge (de-assert /IRQ):
```csharp
// Before
IRQ_enable = false;

// After
IRQ_enable = false;
NesCore.statusmapperint = false; // acknowledge: de-assert /IRQ line
```

#### 3. Main.cs — IRQ polling includes mapper source

```csharp
// Before
irq_pending = (irqPollI == 0 && (statusframeint || statusdmcint));

// After
irq_pending = (irqPollI == 0 && (statusframeint || statusdmcint || statusmapperint));
```
