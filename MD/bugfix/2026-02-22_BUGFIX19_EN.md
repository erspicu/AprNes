# BUGFIX19: DMC DMA Cycle Stealing

**Date**: 2026-02-22
**Baseline**: 169 PASS / 5 FAIL → **171 PASS / 3 FAIL** (+2 PASS)

## Problem

DMC DMA sample fetches do not consume CPU cycles. The `in_tick` guard in `tick()` prevents the recursive `Mem_r()` inside `dmcfillbuffer()` from advancing the PPU, making DMC fetches free (0 cycle stealing).

**Failing tests**:
- `sprdma_and_dmc_dma/sprdma_and_dmc_dma` — All OAM DMA cycles show 399 (missing DMC stolen cycles)
- `sprdma_and_dmc_dma/sprdma_and_dmc_dma_512` — Same
- `dmc_dma_during_read4/dma_2007_read` — Timeout (CRC correct but no "Passed" text)
- `dmc_dma_during_read4/dma_4016_read` — $4016 phantom read not implemented
- `dmc_dma_during_read4/double_2007_read` — Timeout

## Root Causes

### 1. DMC DMA stolen cycles not implemented
`dmcfillbuffer()` calls `Mem_r(dmcaddr)` to read a sample, but the `in_tick` guard causes `tick()` to return immediately. PPU does not advance, CPU is not halted. Real NES DMC DMA steals 3-4 CPU cycles.

### 2. Load vs Reload DMA differences
- **Load DMA** (triggered by $4015 write): schedules halt at GET cycle → **3 cycles normally**, **4 if write-delayed**
- **Reload DMA** (when buffer is exhausted): schedules halt at PUT cycle → **4 cycles normally**, **3 if write-delayed**

### 3. Phantom read rules
While the CPU is halted it repeatedly reads the address currently on the bus. $4016/$4017 only trigger on the halt cycle (NES-001 /OE continuity), other PPU registers trigger on every no-op cycle.

### 4. CRC-only tests lack detection mechanism
dma_2007_read and double_2007_read have multiple valid CRCs due to CPU-PPU sync randomness and do not print "Passed". TestRunner's `--wait-result` cannot find "Passed" text, causing timeout.

## Fixes

### Fix 1: CPU bus state tracking (MEM.cs)
Added `cpuBusAddr` / `cpuBusIsWrite` static fields. Set before the `tick()` call in `Mem_r`, `Mem_w`, `ZP_r`, `ZP_w`, tracking the CPU's current bus state.

### Fix 2: dmc_stolen_tick() (MEM.cs)
Added PPU-only tick function. Advances 3 PPU dots + NMI edge detection + IRQ line tracking. Does not call `apu_step()` (avoids recursion, already called inside apu_step).

### Fix 3: dmcfillbuffer() rewrite (APU.cs)
Load/Reload type-based stolen cycle model:

```
Standalone DMC DMA:
  Load:   haltCycles = cpuBusIsWrite ? 3 : 2  (total 4 or 3)
  Reload: haltCycles = cpuBusIsWrite ? 2 : 3  (total 3 or 4)

DMC during OAM DMA:
  Middle bytes: 2 extra cycles
  Byte 254:    1 extra cycle
  Byte 255:    3 extra cycles
```

Phantom reads: call `mem_read_fun[cpuBusAddr]` on each halt cycle, but $4016/$4017 only on the first halt cycle (simulating NES-001 /OE behavior).

### Fix 4: OAM DMA bus state tracking (PPU.cs)
`ppu_w_4014()` manually sets `cpuBusAddr`/`cpuBusIsWrite` during halt/alignment/write cycles.

### Fix 5: TestRunner --expected-crc (TestRunner.cs)
Added `--expected-crc` parameter accepting comma-separated valid CRCs. Matches 8-digit hex CRC strings found on the nametable during screen stability detection and timeout paths. Added `FindNametableCrc()` method to scan the nametable for CRC strings.

### Fix 6: Test script updates (run_tests.sh, run_tests_report.sh)
Pass `--expected-crc` for CRC-only tests:
- `dma_2007_read.nes`: `--expected-crc 159A7A8F,5E3DF9C4`
- `double_2007_read.nes`: `--expected-crc 85CFD627,F018C287,440EF923,E52F41A5`
