# BUGFIX29 - AccuracyCoin Open Bus & Branch Dummy Reads Fixes

**Date**: 2026-03-04  
**Test ROM**: AccuracyCoin (Page 1: CPU BEHAVIOR, Page 20: CPU BEHAVIOR 2)  
**Result**: OPEN BUS → PASS, BRANCH DUMMY READS → PASS

---

## Fix 1: Branch Dummy Reads (CPU.cs)

**Symptom**: AccuracyCoin Page 20 `BRANCH DUMMY READS` → FAIL 4 / FAIL 5

**Root Cause**:  
On a taken branch, the 6502 performs extra dummy reads:
- **Cycle 3** (taken, no page cross): dummy read from PC+2 (byte following operand)
- **Cycle 4** (page cross): dummy read from wrong-page address `(PC+2_hi | dest_lo)`

AprNes had bare `tick()` calls for both cycles — no actual memory read, so data bus was never updated.

**Fix**: Replaced bare `tick()` with actual `Mem_r()` calls for all 8 branch opcodes (BCC/BCS/BEQ/BMI/BNE/BPL/BVC/BVS):
- Cycle 3: `Mem_r(r_PC)` — dummy read from PC+2
- Cycle 4: `Mem_r((ushort)((ushort1 & 0xFF00) | (r_PC & 0x00FF)))` — wrong-page dummy read

## Fix 2: CPU Open Bus $4020-$5FFF (MEM.cs)

**Symptom**: AccuracyCoin Page 1 `OPEN BUS` → FAIL 1 ("Reading from open bus is not all zeroes")

**Root Cause**:  
Addresses $4020-$5FFF were routed to `MapperObj.MapperR_ExpansionROM()` which returns 0 for all mappers. Real NES hardware leaves the data bus floating at the previous value (`cpubus`).

**Fix**: Merged $4020-$40FF and $4100-$5FFF into a single open bus lambda returning `cpubus` for the entire $4020-$5FFF range.

---

## Fix 3: ZP_r / ZP_w cpubus Update (MEM.cs)

**Symptom**: AccuracyCoin Page 1 `OPEN BUS` → FAIL 4 ("Data bus actually exists, not faked")

**Root Cause**:  
`ZP_r` and `ZP_w` did not update `cpubus`. During read-modify-write instructions on zero page (e.g. `LSR zp,X`), the write cycles should update the data bus. The open bus test jumps to $5600 (open bus) and executes `LSR <$56, X` expecting the write value ($60) to appear on the data bus, making the next fetch return RTS ($60).

**Fix**: Added `cpubus` updates to both functions:
- `ZP_r`: `byte val = NES_MEM[addr]; cpubus = val; return val;`
- `ZP_w`: `NES_MEM[addr] = value; cpubus = value;`

---

## Fix 4: Controller $4016 Open Bus Bits (JoyPad.cs)

**Symptom**: AccuracyCoin Page 1 `OPEN BUS` → FAIL 6 ("Upper 3 bits of controller read are open bus")

**Root Cause**:  
`gamepad_r_4016()` returned `P1_r` directly. `P1_r` had bit 6 hardcoded to 1 (0x40 = released) which happened to pass the simple test but failed when cpubus upper bits ≠ $40 (e.g., when a dummy read sets cpubus to $F0).

**Fix**: Return `(byte)((P1_r & 0x1F) | (cpubus & 0xE0))` — controller data only drives bit 0, upper 3 bits are CPU open bus.

---

## Fix 5: $4015 Read Does Not Update Data Bus (MEM.cs + APU.cs)

**Symptom**: AccuracyCoin Page 1 `OPEN BUS` → FAIL 7 ("Reading $4015 should not update data bus")  
Also covers FAIL 9 ("Bit 5 of $4015 is open bus")

**Root Cause**:  
- `Mem_r` always set `cpubus = val` after every read. But $4015 (APU STATUS) reads internal APU lines — the 2A03 chip doesn't drive the data bus during this read.
- `apu_r_4015()` did not include bit 5 as open bus.

**Fix**:
1. In `Mem_r`: skip cpubus update when `address == 0x4015`
2. In `apu_r_4015()`: add `status |= (byte)(cpubus & 0x20)` for bit 5 open bus behavior
