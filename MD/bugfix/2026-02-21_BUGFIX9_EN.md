## 34. APU Frame Counter Timing: frameReload Offset Fix, IRQ Triple-Assert, Even/Odd Jitter

### Symptom

128 PASS / 46 FAIL. The following 7 APU timing tests all FAIL with "too late" error messages:

| Test ROM | Error |
|----------|-------|
| apu_test/4-jitter | too late |
| apu_test/5-len_timing | too late |
| apu_test/6-irq_flag_timing | too late |
| blargg_apu_2005/04.clock_jitter | too late |
| blargg_apu_2005/05.len_timing_mode0 | too late |
| blargg_apu_2005/06.len_timing_mode1 | too late |
| blargg_apu_2005/07.irq_flag_timing | too late |

### Root Cause

Three overlapping problems all delay frame counter events:

1. **$4017 write offset +7 too large**: should be +2 (plus even/odd jitter)
2. **frameReload4[0] off by 1 cycle**: 7457 should be 7458
3. **Missing IRQ pre-fire and post-fire**: on real NES, mode 0 IRQ flag asserts in the cycle before and after step 3 as well (3 consecutive cycles total)

### Fix

#### 1. APU.cs — Update frameReload Constants

```csharp
// Old:
static int[] frameReload4 = { 7457, 7456, 7458, 7458 };
static int[] frameReload5 = { 7457, 7456, 7458, 7458, 7453 };
static int framectrdiv = 7457;

// New:
static int[] frameReload4 = { 7458, 7456, 7458, 7458 };
static int[] frameReload5 = { 7458, 7456, 7458, 7458, 7452 };
static int framectrdiv = 7458;
```

#### 2. IO.cs — $4017 Write Offset + Even/Odd Jitter

```csharp
// Old:
framectrdiv = frameReload5[0] + 7;
framectrdiv = frameReload4[0] + 7;

// New:
int jitter = 2 + (apucycle & 1); // even/odd CPU cycle jitter
framectrdiv = frameReload5[0] + jitter;
framectrdiv = frameReload4[0] + jitter;
```

NES hardware produces ±1 cycle jitter on $4017 writes based on whether the CPU cycle is even or odd.

#### 3. APU.cs — IRQ Pre-Fire (1 Cycle Before Step 3)

```csharp
// Before the frame counter decrement in apu_step():
if (framectrdiv == 2 && framectr == 3 && ctrmode == 4 && !apuintflag)
    statusframeint = true;
```

`framectrdiv == 2` means the next decrement results in 1 (does not trigger a step), and the one after that triggers step 3. So pre-fire occurs 1 cycle before step 3.

#### 4. APU.cs — IRQ Post-Fire (1 Cycle After Step 3)

Add `irqAssertCycles` countdown counter:

```csharp
static int irqAssertCycles = 0;

// At start of apu_step() (before frame counter decrement):
if (irqAssertCycles > 0 && !apuintflag)
{
    statusframeint = true;
    --irqAssertCycles;
}

// In clockframecounter() when step 3 fires:
if (!apuintflag && framectr == 3 && ctrmode == 4)
{
    statusframeint = true;
    irqAssertCycles = 1; // assert once more next cycle
}
```

Triple-assert result: cycle 29831 (pre-fire) → 29832 (step 3) → 29833 (post-fire).
blargg 6-irq_flag_timing test 4 verifies: reading $4015 at 29832 clears the flag, but the second read can still see the flag (because post-fire re-asserts at 29833).

#### 5. APU.cs — Remove !statusframeint Guard

```csharp
// Old: if (!apuintflag && framectr == 3 && ctrmode == 4 && !statusframeint)
// New: if (!apuintflag && framectr == 3 && ctrmode == 4)
```

Allows the flag to be re-asserted after being cleared by a $4015 read.

### Mode 0 Timing Verification (from $4017 write, even cycle)

```
Pre-fire:  7460 + 7456 + 7458 + 7458 - 1 = 29831  ← flag first set ✓
Step 3:    7460 + 7456 + 7458 + 7458     = 29832  ← flag re-set, length ✓
Post-fire: 29833                                    ← flag re-set ✓
```
