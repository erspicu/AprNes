# BUGFIX47: AccuracyCoin P19 Sprites On Scanline 0

**日期**: 2026-03-08
**基線**: 119/136 → **120/136** (+1)
**修復測試**: P19 Test 5 (Sprites On Scanline 0) err=2 → PASS

## 根因

P19 Test 5 驗證 sprite 是否能正確出現在 scanline 0。

真實 NES PPU 在 pre-render line (261) 的 dots 257-320 會從 secondary OAM 取 sprite tile data，
使用 `(261 & 255) = 5` 作為有效 scanline 進行 in-range 檢查。Secondary OAM 保留前一個
visible scanline (239) 的 evaluation 結果。若 secondary OAM 中有 sprite Y 位置使其在
scanline 5 範圍內，該 sprite 的 tile data 會被載入 shift registers 並持續到 scanline 0。

之前的實作缺少：
1. Secondary OAM buffer 和 per-dot sprite evaluation FSM
2. Pre-render line sprite data 計算
3. Scanline 0 使用 pre-render sprite data 進行 sprite 0 hit detection

## 修復

### PPU.cs — Secondary OAM 和 Per-dot Evaluation

新增 secondary OAM (32 bytes) 和完整的 per-dot sprite evaluation FSM：

- **Dots 1-64**: Clear secondary OAM (寫入 $FF, 每 2 dots 一個 byte)
- **Dot 65**: 初始化 evaluation state (SpriteEvalInit)
- **Dots 65-256**: Per-dot evaluation (SpriteEvalTick) — odd dot 讀 primary OAM, even dot 寫/檢查 secondary OAM
- **Dot 256**: 記錄 evalSpriteCount 和 evalSprite0Visible (SpriteEvalEnd)
- **Pre-render dot 257**: PrecomputePreRenderSprites() — 檢查 secondary OAM 中是否有 sprite 在 scanline 5 範圍內

### PPU.cs — ppu_r_2004()

改為返回 `oamCopyBuffer`（evaluation FSM 最後讀取的 byte），取代之前的近似計算。
Dots 257-320 返回 secondary OAM 對應 byte。

### PPU.cs — PrecomputeSprite0Line()

Scanline 0 若 primary OAM sprite 不在範圍內，使用 `prerender_sprite0` data。

## 注意

Sprite overflow flag 仍由 PrecomputeOverflow() 在精確 cycle 設定，
FSM 中移除了重複的 isSpriteOverflow 設定以避免 blargg 回歸。
