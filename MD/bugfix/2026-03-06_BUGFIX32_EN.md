# BUGFIX32: Load DMA Stolen Cycles — cpuCycleCount Parity

**Date**: 2026-03-06
**Commits**: afe3e17, 86743fe
**Baseline**: 174/174 PASS (maintained)

## Problem

Load DMA (triggered by $4015 write) stolen cycle count was calculated using
`cpuBusIsWrite` as a proxy for GET/PUT phase. This works when reads always
land on GET cycles and writes on PUT cycles, but breaks when reads land on
PUT cycles (e.g., during AccuracyCoin clockslide sequences).

## Root Cause

The 6502 alternates GET/PUT phases every cycle. DMA alignment depends on
which phase the CPU is in when DMA triggers:
- GET phase: halt + dummy + read = 3 stolen cycles
- PUT phase: halt + dummy + align + read = 4 stolen cycles

`cpuBusIsWrite` conflates read/write with GET/PUT. In most code, reads are
on GET and writes on PUT, so it works. But clockslide sequences and other
tight timing loops can have reads on PUT cycles, causing the wrong stolen
cycle count.

## Fix

Load DMA now uses `cpuCycleCount & 1` (Master Clock derived) to determine
GET/PUT phase:

```csharp
if (isLoad)
{
    bool isPutCycle = (cpuCycleCount & 1) != 0;
    haltCycles = isPutCycle ? 3 : 2;
}
```

Reload DMA retains `cpuBusIsWrite` proxy (changing it causes sprdma_and_dmc_dma
regression — Reload DMA fires at different phase than Load DMA).
