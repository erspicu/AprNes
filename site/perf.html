關鍵技術發現

1. Debug JIT 的特殊性

AprNes 核心使用 .NET Framework Debug 組態，JIT 最佳化等級遠低於 Release。許多在 Release 由 JIT 自動完成的工作（loop unrolling、register allocation、inlining）必須手動完成。

所有欄位均為 static，JIT 無法 enregister，每次讀取都是記憶體 load。「local shadow」技法（在函式開頭 copy static 至 local）可將多次 static read 降為 1 次。

2. 函式指標化 dispatch 的邊際效應（P9 +7.6%，P24 +1.1%）

256-case switch → Action[] 的改善遠超預期（+7.6%），因為 Debug JIT 不生成完整 jump table。進一步升級至 delegate*<void>[]（原生函式指標）再獲 +1.1%，但效益因 Mem_r()/tick() 的固有開銷而有上限。同樣機制應用於 mem_read_fun（P25）卻退步 -4.8%，原因是 Mapper instance method 需要 static wrapper，抵消了 dispatch 節省。

// P9: switch → Action[]
opHandlers[opcode]();      // delegate call: ~5 loads + null check

// P24: Action[] → delegate*<void>[]
opFnPtrs[opcode]();        // 1 load + 1 calli — C-style function table

3. 移除冗餘 branch 的意外收益（P6 +6.1%）

RenderBGTile() 內層 8-pixel 迴圈中的 if (screenX > 255) break; 看似無害，但呼叫方已有邊界 guard。移除後 JIT 能對整個迴圈做更積極的 unrolling，效益遠超預期（+6.1%）。任何內層迴圈中的非必要 conditional exit 都值得審視。

4. 快取新增欄位的 cache line 成本

多次嘗試「增加快取欄位來省略計算」均以退步告終（P13 -2.2%，P27 -1.5%，tile fetch -1.2%）。根本原因：新欄位改變物件 field layout，使熱路徑資料跨越更多 cache line，增加 cache miss。在 .NET Debug JIT 下，欄位位置對效能的影響往往超過演算法節省。

5. 測試方法與 CPU 熱降頻

連續 benchmark 會導致 CPU 熱降頻，FPS 可虛假偏低 10% 以上。每次測試前 sleep 60 必不可少。第一次 run 結果往往比 run2/run3 低 3–5%（CPU 從冷啟動暖機），建議以多次平均為準。
