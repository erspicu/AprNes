### 2. Multi-Key Rollover Investigation → Confirmed Not an Emulator Issue

**Symptom**: In SMB3, Z+X+→ cannot be triggered simultaneously, but Z+X+← can.

**Investigation process**:
- Analyzed keyboard input chain (ProcessCmdKey → NES_KeyMAP → P1_ButtonPress)
- Attempted to switch to per-frame polling via GetAsyncKeyState
- Built `KeyTest/KeyTest.exe` to test directly with GetAsyncKeyState

**Conclusion**: GetAsyncKeyState also fails to detect Z+X+→, confirming **hardware keyboard Matrix Ghosting** — unrelated to the emulator. Reverted to the original event-driven implementation.

**Document**: `bugfix/2026-02-25_KEYBOARD_GHOSTING.md`

---

### 3. FPS Limiter Accuracy Improvement (PPU.cs / Main.cs)

**Problems**:
1. Target time `0.016s` = 60.0 Hz; NES NTSC correct frequency is 60.0988 Hz
2. `Thread.Sleep(1)` overshoots on the final sleep, causing ±1ms error per frame
3. Per-frame independent timing (StopWatch.Restart) with no accumulated error compensation

**Fixes**:
- Target changed to `1.0 / 60.0988 ≈ 0.016639s`
- Hybrid sleep: use Sleep(1) when >1ms from deadline, switch to spin-wait for the final <1ms
- `_fpsDeadline` advances cumulatively, automatically compensating for single-frame drift

Accuracy: ±1ms → <0.1ms

---

### 4. FPS Display Statistics Improvement (PPU.cs / AprNesUI.cs)

**Problems**:
- `frame_count` was not `volatile`; JIT may read a stale value
- Read + clear-to-zero was not atomic, causing a race condition
- WinForms Timer does not guarantee an exact 1000ms interval, introducing denominator error
- Integer display cannot distinguish subtle differences

**Fixes**:
- Added `volatile` to `frame_count`
- Switched to `Interlocked.Exchange` for atomic read + clear
- Used `Stopwatch` to measure the actual tick interval as the denominator
- Display changed to one decimal place (e.g. `fps : 60.1`)
