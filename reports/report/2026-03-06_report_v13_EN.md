### 1. Load DMA countdown changed to parity-dependent

**Goal**: Fix 3 long-standing FAIL blargg tests to achieve a perfect 174/174 pass rate.

**Problem**: The Load DMA ($4015 write trigger) scheduling delay was hardcoded at 3 APU cycles,
but real hardware (and Mesen2) uses CPU cycle parity to determine whether the delay is 2 or 3.

**Mesen2 reference** (`DeltaModulationChannel.cpp` SetEnabled):
```cpp
if((_console->GetCpu()->GetCycleCount() & 0x01) == 0) {
    _transferStartDelay = 2;
} else {
    _transferStartDelay = 3;
}
```

**Fix** (`APU.cs`):
```csharp
// Old: dmcLoadDmaCountdown = 3;
// New:
dmcLoadDmaCountdown = (apucycle & 1) != 0 ? 2 : 3;
```

Mapping: `apucycle` odd = Mesen2 CycleCount even → delay 2;
`apucycle` even = Mesen2 CycleCount odd → delay 3.

**Tests fixed**:

| Test | Old Result | New Result |
|------|------------|------------|
| dmc_dma_during_read4/double_2007_read | FAIL | PASS |
| read_joy3/count_errors | FAIL | PASS |
| read_joy3/count_errors_fast | FAIL | PASS |

**Result**: 171 PASS → **174 PASS / 0 FAIL / 174 TOTAL**

### 2. AccuracyCoin test 82 (IFlagLatency) investigation

**Problem**: AccuracyCoin test 82 hangs and never completes.

**Root cause analysis**:
The DMC DMA stolen cycle calculation uses `cpuBusIsWrite` as an approximation for GET/PUT phase.
In AccuracyCoin's clockslide sequence, 9 out of 79 DMAs fall on JSR push (write cycles),
each undercounting by 1 stolen cycle, accumulating a drift of 8 cycles and preventing the DMA
from aligning to the $4013 opcode fetch.

**Attempts and results**:

| Approach | Result | Problem |
|----------|--------|---------|
| Deferred Reload DMA (delay all by 1 cycle) | 172/174 | Breaks 2 sprdma tests |
| Write-cycle only deferral | 173/174 | Breaks sprdma_512 |
| Reload haltCycles=3 always | 172/174 | Breaks sprdma |
| apucycle parity for Reload | 172/174 | Breaks sprdma |
| Opposite parity | 168/174 | Breaks 6 tests |
| Unified parity (no Load/Reload distinction) | 168/174 | Breaks 6 tests |
| Global cpuCycleCount counter | 172/174 | Same result as apucycle |

**Conclusion**: A proper fix requires true GET/PUT phase tracking (replacing `cpuBusIsWrite` with
`apucycle` parity), but currently both parity directions break sprdma tests. The root cause is
that `apucycle`'s initial offset does not match the real hardware phase. This requires further
calibration of the initial offset and is deferred to future work.

**Architecture assessment**: The sub-cycle timing change itself is small (~15 lines in APU.cs),
but the parity offset calibration issue means a naive application causes regressions. Required steps:
1. Introduce a dedicated `cpuCycleCount` counter (separate from apucycle)
2. Calibrate the initial phase so GET/PUT aligns with real hardware
3. Verify that sprdma + AccuracyCoin pass simultaneously
