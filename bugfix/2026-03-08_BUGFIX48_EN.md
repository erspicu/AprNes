# BUGFIX48: AccuracyCoin P19 $2004 Stress Test

**Date**: 2026-03-08
**Baseline**: 120/136 → **121/136** (+1)
**Fixed Test**: P19 Test 6 ($2004 Stress Test) err=1 → PASS

## Root Cause

The P19 $2004 Stress Test reads $2004 at every dot (0-340) on scanline 128, collecting
complete data across 21 frames, then verifies that each dot's return value matches the
behavior of the PPU sprite evaluation state machine.

Four issues:

### 1. ppu_r_2004 range boundaries not compensated for tick-before-read timing offset

Our tick() model processes 3 PPU dots before a bus read. When $2004 is read at cx=N+1,
it corresponds to hardware dot N. The range checks in ppu_r_2004 need a +1 offset to compensate.

### 2. Clear phase returns oamCopyBuffer instead of hardcoded $FF

During the dots 1-64 secondary OAM clear phase, $2004 should always return $FF,
regardless of oamCopyBuffer.

### 3. Attribute byte masking applied at wrong location

Hardware masks bits 2-4 of the attribute byte on the internal bus when reading from
primary OAM. The previous implementation applied masking when writing to secondary OAM,
causing $2004 reads to return unmasked raw values. The test expects both odd and even
dots to return the masked value (e.g., $61 instead of $7D).

### 4. Post-evaluation phase missing secondary OAM read and address advance

After all 64 sprites have been checked (oamCopyDone), even dots should read from
secondaryOAM[secOAMAddr] into oamCopyBuffer (writes become reads), and spriteEvalAddrH
should continue advancing to cycle through primary OAM.

## Fix

### PPU.cs — ppu_r_2004()

Range boundary +1 compensation:
- Clear phase: cx [2, 65] (was [1, 64]), returns hardcoded $FF
- Eval phase: cx [66, 257] (was [65, 256])
- Sprite tile fetch: cx [258, 321] (was [257, 320])

### PPU.cs — SpriteEvalTick()

When reading primary OAM on an odd dot, if spriteEvalAddrL == 2 (attribute byte),
mask immediately: `oamCopyBuffer &= 0xE3`

### PPU.cs — SpriteEvalWrite()

Post-eval phase (oamCopyDone && secOAMAddr < 0x20):
- Read from secondaryOAM[secOAMAddr] into oamCopyBuffer
- Advance spriteEvalAddrH (continue cycling through primary OAM addresses)

## Verification

- blargg: 174/174 (no regression)
- AccuracyCoin: 121/136 (+1, P19 all 6 subtests PASS)
