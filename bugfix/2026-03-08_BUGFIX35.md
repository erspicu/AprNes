# BUGFIX35: Arbitrary Sprite Zero + Misaligned OAM + Parallel Test Runner + Debug Log Fix

## 日期: 2026-03-08

## 基線: 174 PASS (舊測試), 108/136 AccuracyCoin PASS (BUGFIX34 後)

## 修復內容

### 1. Arbitrary Sprite Zero (PPU.cs) — AccuracyCoin $0458 PASS
- **問題**: PrecomputeSprite0Line() 硬編碼從 OAM[0] 讀取 sprite 0 資料
- **正確行為**: NES 硬體在 dot 65 開始 sprite evaluation，使用當前 OAMADDR 作為起始地址。第一個被處理的 sprite 視為 "sprite 0"，即使它不在 OAM index 0
- **修復**:
  - 新增 `sprite0_eval_addr` 欄位
  - 在 dot 65（visible scanlines + pre-render scanline 261）儲存當前 `spr_ram_add`
  - PrecomputeSprite0Line() 改用 `sprite0_eval_addr` 計算 addrH/addrL，依序讀取 Y/tile/attr/X
  - 正確模擬 sprite evaluation 的 4-byte wrapping（addrL 0→1→2→3→0 時 addrH++）

### 2. Misaligned OAM Behavior (PPU.cs) — AccuracyCoin $045A PASS
- **問題**: 同上，且未處理 OAMADDR 非 4-byte 對齊的情況
- **正確行為**: 當 OAMADDR 不是 4 的倍數時（如 $81），sprite evaluation 仍從該地址開始讀取 Y coordinate，addrL 從非零值開始遞增
- **修復**: 與上面相同的修復自動涵蓋了此場景
- **注意**: sprite0_eval_addr 保留完整地址（不做 & 0xFC 對齊），addrH/addrL 分別計算

### 3. Debug Log PID-based Path (CPU.cs)
- **問題**: 所有 headless 模式的 emulator instance 寫入同一個 `emu_debug.log`，導致平行執行時 IOException（檔案被佔用）
- **修復**: dbgInit() 改用 `%TEMP%/aprnes_debug_{PID}.log`，除非 TestRunner 明確設定 DebugLogPath

### 4. Python Parallel Test Runner (run_tests.py)
- **新增**: `run_tests.py` — 使用 ThreadPoolExecutor 平行執行 174 個測試 ROM
- **效果**: 2m11s → ~19s（10 threads，約 6.7x 加速）
- **用法**: `python run_tests.py` (預設 10 threads), `python run_tests.py -j 4` (4 threads)
