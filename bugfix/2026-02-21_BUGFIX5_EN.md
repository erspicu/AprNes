## 16. $2004 OAM Read Modifies spr_ram_add

**Commit**: `e5c7486`

### Symptom

After reading $2004 (OAM Data), subsequent OAM write positions are shifted, causing garbled sprite display.
blargg `sprite_ram.nes` test fails.

### Root Cause

The original `ppu_r_2004()` implementation used post-increment `spr_ram[spr_ram_add++]`,
advancing the OAM pointer address with every read. NES hardware's $2004 read is **non-destructive**
and must not modify the OAM address set by $2003.

Additionally, the unimplemented bits (bits 2–4) of attribute bytes (OAM offset % 4 == 2)
must always return 0 (masked with `& 0xE3`); the original implementation omitted this mask.

### Fix

```csharp
// Before
static byte ppu_r_2004() { return spr_ram[spr_ram_add++]; }

// After
static byte ppu_r_2004()
{
    byte val = spr_ram[spr_ram_add];  // no increment
    if ((spr_ram_add & 3) == 2) val &= 0xE3;  // mask unimplemented attribute bits
    return openbus = val;
}
```

## 17. Sprite Y Evaluation off-by-1 (Incorrect Scan Condition)

**Commit**: `e5c7486`

### Symptom

A row of garbage pixels appears at the top of sprites; the bottom row is missing.

### Root Cause

The Pass 1 Y range scan condition `scanline <= raw_y || scanline - raw_y > height + 1`
is incorrect at boundary cases. The correct visible range should be `[raw_y, raw_y + height]`,
i.e. `scanline >= raw_y && scanline - raw_y <= height`.

### Fix

```csharp
// Before
if (scanline <= raw_y || scanline - raw_y > height + 1) continue;

// After
if (scanline < raw_y || scanline - raw_y > height) continue;
```

## 18. OAM Scanned While Rendering Off Causes False Overflow

**Commit**: `e5c7486`

### Symptom

When a game disables rendering (`ShowBackGround = false` and `ShowSprites = false`),
the sprite overflow flag is still incorrectly set.

### Root Cause

The Pass 1 OAM scan loop in `RenderSpritesLine()` still executes when rendering is disabled,
computes sprite counts, and sets `isSpriteOverflow`. NES hardware does not perform sprite
evaluation when rendering is disabled.

### Fix

```csharp
// After: only scan OAM when rendering is enabled
bool renderingEnabled = ShowBackGround || ShowSprites;
if (renderingEnabled)
{
    for (int oam_th = 0; oam_th < 64; oam_th++) { ... }
}
```

## 19. VBL Suppress Check Uses Wrong Scanline Number

**Commit**: `e5c7486`

### Symptom

`ppu_r_2002()` should suppress the VBL flag when read on the same cycle it is set,
but the suppress condition uses the wrong scanline number.

### Root Cause

In the cycle-accurate PPU (`ppu_step_new`), VBL is set at scanline 241 cycle 1.
The original suppress condition checked `scanline == 240` (corresponding to the timing
of the old `ppu_step`), which cannot match correctly in cycle-accurate mode.

### Fix

```csharp
// Before
if (ppu_cycles_x == 1 && scanline == 240) SuppressVbl = true;

// After
if (ppu_cycles_x == 1 && scanline == 241) SuppressVbl = true;
```

## 20. NMI Still Fires When VBL Is Suppressed

**Commit**: `a289801`

### Symptom

blargg `nmi_suppression.nes` test fails. When a `$2002` read suppresses the VBL flag,
NMI still fires.

### Root Cause

In `ppu_step_new()`, the VBL set logic at scanline 241 cycle 1 sets `nmi_pending`
regardless of the `SuppressVbl` flag:

```csharp
// Before
isVblank = true;
if (NMIable) nmi_pending = true;
SuppressVbl = false;
```

NES hardware specification: if VBL is suppressed ($2002 read on the same cycle),
`isVblank` must not be set and NMI must not fire.

### Fix

```csharp
// After
if (!SuppressVbl)
{
    isVblank = true;
    if (NMIable) nmi_pending = true;
}
SuppressVbl = false;
```

## 21. $2000 NMI Edge Trigger Not Implemented

**Commit**: `a289801`

### Symptom

blargg `nmi_disable.nes` and `nmi_control.nes` tests fail.
When NMI is enabled during VBL by writing `$2000` (bit 7: 0→1),
NES hardware should immediately fire NMI (rising-edge detection), but the emulator does nothing.

### Root Cause

`ppu_w_2000()` only records the new value of `NMIable` without detecting the rising edge:

```csharp
// Before
NMIable = ((value & 0x80) > 0);
```

### Fix

```csharp
// After
bool wasNMIable = NMIable;
NMIable = ((value & 0x80) > 0);
// Rising edge: enabling NMI during VBL → fire immediately
if (!wasNMIable && NMIable && isVblank) nmi_pending = true;
```

## 22. Sprite 0 Hit Changed to Per-Pixel Cycle-Accurate Detection

**Commit**: `a289801`

### Symptom

blargg `sprite_hit 09.timing_basics.nes` and `10.timing_order.nes` tests fail.
Sprite 0 Hit flag is set at the wrong time.

### Root Cause

The original implementation batch-detected sprite 0 hit at cycle 257 (after scanline ends),
deciding all at once whether a collision occurred anywhere on the line. NES hardware's sprite 0 hit
is detected **per-pixel** and the flag is set immediately at the PPU cycle where the BG pixel renders.

### Fix

Add `PrecomputeSprite0Line()` to pre-compute sprite 0 tile data at the start of each scanline,
then detect collisions in real time in the per-pixel loop of `RenderBGTile()`:

```csharp
// Pre-compute at start of each scanline
static void PrecomputeSprite0Line()
{
    sprite0_on_line = false;
    if (isSprite0hit || !ShowBackGround || !ShowSprites) return;

    int y_loc = spr_ram[0] + 1;
    int height = Spritesize8x16 ? 15 : 7;
    if (scanline < y_loc || scanline - y_loc > height) return;
    // read sprite 0 tile data...
    sprite0_on_line = true;
}

// Real-time detection during BG render
static void RenderBGTile()
{
    // ... render BG pixels ...
    if (sprite0_on_line && !isSprite0hit && screenX != 255)
    {
        // check if both sprite 0 pixel and BG pixel are opaque
        if (sprPixel != 0 && bgPixel != 0)
            isSprite0hit = true;
    }
}
```

## 23. APU Init/Reset State Incorrect

**Commit**: `031627c`

### Symptom

Multiple blargg APU tests fail:
- `apu_reset/4015_cleared.nes`: $4015 should be $00 after reset
- `apu_reset/len_ctrs_enabled.nes`: length counters should be disabled after reset
- `apu_reset/irq_flag_cleared.nes`: IRQ flag should be cleared after reset
- `apu_reset/works_immediately.nes`: reset should take effect immediately

### Root Cause

`initAPU()` and the soft reset function do not correctly set the initial state:

1. **Power-on**: $4015 should be $00 (all channels disabled), $4017 should be $00 (mode 0)
2. **Soft reset**: should preserve ctrmode and apuintflag, clear IRQ flags, disable all channels
3. **$4017 write**: mode 1 should immediately clock length/envelope/sweep
4. **$4010 write**: clearing DMC IRQ enable should also clear the DMC IRQ flag

### Fix

**APU.cs** — `initAPU()` completely rewritten, `apuSoftReset()` added:

```csharp
static void initAPU()
{
    // $4015 = $00: all channels disabled
    apustatus = 0;
    // $4017 = $00: mode 0, IRQ enabled
    ctrmode = false; apuintflag = false;
    // All length counters, envelopes, sweeps zeroed
    // ...
}

static void apuSoftReset()
{
    // Preserve ctrmode, apuintflag
    // Clear all IRQ flags, disable channels
    apustatus = 0;
    // ...
}
```

**IO.cs** — $4017 write fix, $4010 DMC IRQ clear:

```csharp
// $4017: mode 1 immediate clock
if (ctrmode) { /* immediately clock quarter/half frame */ }

// $4010: bit 7 controls DMC IRQ
if ((value & 0x80) == 0) dmcirqflag = false;  // clear DMC IRQ
```

## 24. MMC1 CHR Bank Out-of-Bounds Access

**Commit**: `f3ab0e8`

### Symptom

ROMs using the MMC1 mapper (e.g. blargg_nes_cpu_test5) crash or show garbage during CHR bank switching.

### Root Cause

`Mapper001.cs`'s `MapperR_CHR()` does not apply modulo to the bank selection value.
When a game sets a CHR bank value exceeding the actual number of CHR-ROM banks,
the array index goes out of bounds:

```csharp
// Before (may go out of bounds)
if (CHR_Bankmode > 0)
{
    if (address < 0x1000) return CHR_ROM[address + (CHR0_Bankselect << 12)];
    else return CHR_ROM[(address - 0x1000) + (CHR1_Bankselect << 12)];
}
else return CHR_ROM[address + 0x2000 * (CHR0_Bankselect >> 1)];
```

### Fix

```csharp
// After: add modulo to prevent out-of-bounds
if (CHR_Bankmode > 0)
{
    int banks4k = CHR_ROM_count * 2;
    if (address < 0x1000) return CHR_ROM[address + ((CHR0_Bankselect % banks4k) << 12)];
    else return CHR_ROM[(address - 0x1000) + ((CHR1_Bankselect % banks4k) << 12)];
}
else return CHR_ROM[address + 0x2000 * ((CHR0_Bankselect >> 1) % CHR_ROM_count)];
```

## 25. MMC1 One-Screen Mirroring Not Supported

**Commit**: `f3ab0e8`

### Symptom

blargg_nes_cpu_test5 (MMC1 mapper) shows a completely black screen. CPU runs normally but nothing appears on screen.

### Root Cause

MMC1 supports 4 nametable mirroring modes:
- 0: one-screen, lower bank
- 1: one-screen, upper bank
- 2: vertical
- 3: horizontal

The original `Mapper001.cs` only handled vertical and horizontal, ignoring one-screen modes.
MMC1 power-on defaults to mirroring mode 0 (one-screen lower), and `MEM.cs`'s nametable
write function did not support one-screen.

Additionally, `PRG_Bankmode` was initialized to 0, but the NES hardware MMC1 power-on default
should be 3 (fix last bank at $C000), causing incorrect PRG bank calculation.

### Fix

**Mapper001.cs** — support all 4 mirroring modes, fix PRG initial value:

```csharp
// Before
int PRG_Bankmode = 0;
// ...
if (Mirroring_type == 2) *Vertical = 1;
else if (Mirroring_type == 3) *Vertical = 0;

// After
int PRG_Bankmode = 3; // NES hardware power-on default
// ...
if (Mirroring_type == 0) *Vertical = 2;      // one-screen, lower bank
else if (Mirroring_type == 1) *Vertical = 3;  // one-screen, upper bank
else if (Mirroring_type == 2) *Vertical = 1;  // vertical
else *Vertical = 0;                            // horizontal
```

**MEM.cs** — nametable writes support one-screen:

```csharp
int mirror = *Vertical;
if (mirror >= 2)
{
    // One-screen: map all 4 nametables to the same 1KB
    int rel = _vram_addr_wrap & 0x3FF;
    ppu_ram[0x2000 + rel] = ppu_ram[0x2400 + rel] =
    ppu_ram[0x2800 + rel] = ppu_ram[0x2C00 + rel] = val;
}
else if (mirror == 1) { /* vertical mirroring */ }
else { /* horizontal mirroring */ }
```

## 26. Sprite Overflow and Rendering Y+1 Offset Not Separated

**Commit**: `f3ab0e8`

### Symptom

After fixing the sprite Y+1 display offset, sprite_overflow_tests/2.Details.nes
regressed from PASS to FAIL (sub-test #5 fails).

Sub-test #5 verifies: "9 sprites at Y=239 should set the overflow flag."
Sub-test #6 verifies: "Y=240 should not set the overflow flag."

### Root Cause

NES PPU sprite processing has two independent stages using different Y baselines:

1. **Sprite evaluation** (overflow counting): uses the raw OAM Y value (`raw_y`).
   Hardware scans OAM on the current scanline to determine which sprites are in range.

2. **Sprite rendering** (pixel drawing): uses Y+1 (OAM Y + 1).
   NES hardware has a one-scanline pipeline delay — sprites evaluated on scanline N
   are rendered on scanline N+1.

The previous fix uniformly applied Y+1 to both evaluation and rendering, causing sprites
with OAM Y=239 to be judged as Y=240 (out of range) during overflow evaluation and not triggering overflow.

### Fix

Separate overflow counting from rendering selection in `RenderSpritesLine()` Pass 1:

```csharp
for (int oam_th = 0; oam_th < 64; oam_th++)
{
    int oam_y = spr_ram[oam_th << 2];

    // Overflow evaluation: use raw OAM Y (hardware evaluation timing)
    if (scanline >= oam_y && scanline - oam_y <= height)
    {
        if (++spriteCount == 9) isSpriteOverflow = true;
    }

    // Rendering selection: use Y+1 (pipeline delay)
    int render_y = oam_y + 1;
    if (scanline < render_y || scanline - render_y > height) continue;
    if (selCount < 8) sel[selCount++] = oam_th;
}
```

**Behavioral difference**:

| OAM Y | Scanline | Overflow Eval (raw Y) | Rendering Selection (Y+1) |
|-------|----------|----------------------|---------------------------|
| 239   | 239      | 239-239=0 ✓ counted  | 239-240=-1 ✗ not selected |
| 100   | 101      | 101-100=1 ✓ counted  | 101-101=0 ✓ selected      |
| 100   | 108      | 108-100=8 ✗ not counted | 108-101=7 ✓ selected   |

Sprites with OAM Y=239 are counted for overflow (on scanline 239) but not selected for rendering (Y+1=240 exceeds visible range). This correctly matches NES hardware behavior.

### Impact

- sprite_overflow_tests/2.Details.nes: PASS (regression fixed)
- All 8 sprite_hit_tests: PASS (no regression)
- Other sprite_overflow_tests (1.Basics, 5.Emulator): PASS (no regression)

## Additional Fixes

### CPU open bus ($4020-$40FF)

**Commit**: `f3ab0e8`

`MEM.cs`: The $4020-$40FF range is an unused area on NES hardware.
Reads should return the residual value on the CPU data bus (open bus); writes have no effect.

```csharp
// Added open bus handling
else if (address < 0x4100) mem_write_fun[address] = (addr, val) => { };
else if (address < 0x4100) mem_read_fun[address] = (addr) => cpubus;
```

Also added `cpubus` variable to track the last data value read/written by the CPU:

```csharp
static byte Mem_r(ushort address)
{
    byte val = mem_read_fun[address](address);
    cpubus = val;
    return val;
}
static void Mem_w(ushort address, byte value)
{
    cpubus = value;
    mem_write_fun[address](address, value);
}
```

### OAM DMA Changed to Use Mem_r()

**Commit**: `f3ab0e8`

`PPU.cs`: `ppu_w_4014()` previously read `NES_MEM[]` directly; changed to read through `Mem_r()`
to ensure correct bus behavior during DMA (triggering mapper side effects, updating cpubus):

```csharp
// Before
for (int i = 0; i < 256; i++) spr_ram[spr_ram_add++] = NES_MEM[oam_address++];

// After
for (int i = 0; i < 256; i++) spr_ram[spr_ram_add++] = Mem_r((ushort)(oam_address++));
```
