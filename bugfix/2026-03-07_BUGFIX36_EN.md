# BUGFIX36: OAM Corruption

## Date: 2026-03-07

## Baseline: 174 PASS (legacy tests), 110/136 AccuracyCoin PASS (after BUGFIX35)

## Fix Details

### OAM Corruption (PPU.cs) — AccuracyCoin $047B PASS
- **Problem**: When the NES PPU rendering transitions from enabled to disabled, it marks OAM rows as corrupted based on the current PPU dot. When rendering is re-enabled (or at frame start), the first 8 bytes of OAM row 0 are copied to each marked row, causing sprites to disappear or duplicate. AprNes had no implementation of this behavior at all.
- **Correct behavior** (reference: Mesen2 NesPpu.cpp):
  - Dots 0-63 (Secondary OAM clear): marks 1 row every 2 dots (`corruptOamRow[dot >> 1]`)
  - Dots 256-319 (Sprite tile fetch): 8-dot segments; first 3 dots increment the row, last 5 dots mark the next row
  - After marking, when rendering is re-enabled or at frame start, copy OAM[0:7] to each marked row
- **Fix**:
  - Added `bool[] corruptOamRow = new bool[32]` and `bool prevRenderingEnabled`
  - `SetOamCorruptionFlags()`: calculates which row to mark based on ppu_cycles_x
  - `ProcessOamCorruption()`: iterates over 32 rows, copies OAM[0:7] to each marked row
  - `ppu_w_2001()`: detects rendering enabled↔disabled transitions, calls the corresponding method
  - Frame start (scanline 262→0): if rendering is enabled, calls `ProcessOamCorruption()`
