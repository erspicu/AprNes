## Background: Cycle-Accurate PPU Rewrite (commit 24687f0)

To fix the CHR bank timing issue where the SMB3 mushroom sprite penetrates "?" blocks, the old batch rendering
(outputting a whole line of pixels at once at scanline cycle 254) was replaced with cycle-accurate per-8-cycle tile fetching:

- **BG tile fetch timing**: Each group of 8 PPU cycles (phases 0–7); CHR address set at phases 4/6, CHR data read at phases 5/7, rendered and loaded into shift register at phase 7.
- **16-bit shift register** (`lowshift`/`highshift`): high byte = current tile, low byte = next tile.
- **A12 detection**: switched to using actual CHR address bit 12 to trigger the MMC3 IRQ (later adjusted again due to timing issues; see Bug 14).
- **Sprite priority for Mapper 003/004**: correct CHR bank read timing makes sprite-vs-background occlusion match NES hardware behavior.

The rewrite introduced Bugs 12, 13, and 14, all fixed in subsequent commits.

## 10. UpdateVramRegister Condition Does Not Include ShowSprites

**Commit**: `718b085`

### Symptom
In scenes where only sprite rendering is enabled (`ShowSprites=true`) but background rendering is disabled (`ShowBackGround=false`),
the VRAM address (Y increment + hori copy) is not updated, causing incorrect scroll addresses for subsequent scanlines and abnormal scrolling.

### Root Cause
In `PPU.cs`'s `ppu_step()`, the condition for calling `UpdateVramRegister()` at cycle 256
(which performs Y increment and hori copy) only checks `ShowBackGround`:

```csharp
// Before
else if (ppu_cycles_x == 256 && ShowBackGround) UpdateVramRegister();
```

NES hardware specification: the PPU performs Y increment at cycle 256 and hori copy at cycle 257 as long as **either** rendering is enabled (background **or** sprites). The condition was missing `ShowSprites`.

### Affected Location
`PPU.cs`: `UpdateVramRegister()` call condition at cycle 256 in `ppu_step()`.

### Fix
```csharp
// After
else if (ppu_cycles_x == 256 && (ShowBackGround || ShowSprites)) UpdateVramRegister();
```

## 11. Sprite Priority Check Uses Stale Buffer_BG_array When Background Is Off

**Commit**: `718b085`

### Symptom
When `ShowBackGround=false` (background rendering disabled), priority=1 (behind-background) sprites should be visible at all positions (since there are no background pixels to occlude them), but some sprites actually disappeared.

### Root Cause
```csharp
// Before
if (pixel != 0 && (Buffer_BG_array[array_loc] == 0 || !priority))
    ScreenBuf1x[array_loc] = ...;
```

When `ShowBackGround=false`, `Buffer_BG_array` still contains data from the **previous frame** (or all zeros, depending on whether background was ever filled). NES hardware specification: when background is off, background pixels are treated as fully transparent (pixel=0), so priority=1 sprites should be drawn unconditionally without consulting `Buffer_BG_array`.

### Affected Location
`PPU.cs`: sprite output condition in `RenderSpritesLine()`.

### Fix
```csharp
// After: add !ShowBackGround; when background is off, ignore Buffer_BG_array
if (pixel != 0 && (!ShowBackGround || Buffer_BG_array[array_loc] == 0 || !priority))
    ScreenBuf1x[array_loc] = ...;
```

Complete decision logic:
| Condition | Result |
|-----------|--------|
| pixel = 0 | Don't draw (transparent pixel) |
| Background off (`!ShowBackGround`) | Draw sprite directly |
| Background pixel transparent (`Buffer_BG_array == 0`) | Draw sprite directly |
| priority = 0 (foreground sprite) | Draw sprite directly |
| priority = 1 (behind sprite) and background opaque | **Don't draw** (occluded by background) |

## 12. Cycle-Accurate PPU: BG Attribute Ring Buffer Read/Write Count Mismatch Per Scanline

**Commit**: `be3f979`

### Symptom
After the cycle-accurate PPU implementation, massive graphical corruption: background tile palettes (attributes) completely wrong, colors a total mess, worst in the lower half of the screen, and the problem accumulates and worsens over time.

### Root Cause
The original implementation used a 4-slot ring buffer (`bg_at_queue[4]`) to delay BG tile attributes by 2 fetch groups:

```
Phase 3 write: bg_at_queue[bg_at_q_wr++ & 3] = ATVal
Phase 7 read:  bg_at_queue[bg_at_q_rd++ & 3]
```

**Writes per scanline ≠ reads per scanline**:

| Section | Event | Count |
|---------|-------|-------|
| Cycles 0–255 (32 tiles) | Phase 3 write | **32** |
| Cycles 320–335 (2 prefetch tiles) | Phase 3 write | **2** |
| Phase 7 render (only when cycle < 256) | Phase 7 read | **32** |
| **Total** | 34 writes, 32 reads | **+2 offset/scanline** |

After each scanline `q_wr - q_rd` increases by 2. Once the difference exceeds 4 (ring buffer size), subsequent reads get **stale attributes from several scanlines ago**, causing systematic palette corruption.

**Concrete trace (after scanline 0)**:
- `q_wr = 36`, `q_rd = 32` (difference = 4, ring buffer entirely filled with stale data)
- First read on scanline 1: `q_rd=32 → slot 32&3=0` → reads tile32_attr (prefetch data), not scan1_tile0_attr

### Affected Location
`PPU.cs`: `bg_at_queue`, `bg_at_q_wr`, `bg_at_q_rd` fields, and
`ppu_rendering_tick()` case 3 and `RenderBGTile()`.

### Fix
Replace the ring buffer with a **3-stage pipeline**:

```csharp
// Before: ring buffer (unbalanced reads/writes per scanline)
static byte[] bg_at_queue = new byte[4];
static int bg_at_q_wr = 0, bg_at_q_rd = 0;

// After: 3-stage pipeline (self-aligning, no index offset issues)
static byte bg_attr_p1 = 0, bg_attr_p2 = 0, bg_attr_p3 = 0;
```

**Phase 3** (unconditional, runs for every tile fetch group):
```csharp
bg_attr_p3 = bg_attr_p2; bg_attr_p2 = bg_attr_p1; bg_attr_p1 = ATVal;
```

**Phase 7 render**:
```csharp
byte renderAttr = bg_attr_p3;  // attribute read 2 fetch groups ago (corresponds to shift reg high byte)
byte nextAttr   = bg_attr_p2;  // attribute read 1 fetch group ago (corresponds to shift reg low byte)
```

**Correctness verification (scanline 0 example)**:
1. Previous scanline prefetches tile0 (phase 3) → `p1=attr0`
2. Previous scanline prefetches tile1 (phase 3) → `p3=old, p2=attr0, p1=attr1`
3. Current scanline cycle 3 (fetch tile2) → `p3=attr0, p2=attr1, p1=attr2`
4. Current scanline cycle 7 renders tile0: `renderAttr=p3=attr0` ✓, `nextAttr=p2=attr1` ✓

The pipeline is self-correcting at scanline boundaries: the pre-render scanline (261)
cycles 0–255 write 32 rounds of noise, but the subsequent 2 prefetch writes at cycles 320–335
correctly overwrite p2/p3, giving the next visible scanline the correct attributes.

## 13. Cycle-Accurate PPU: scanline = -1 Initial Value Causes ScreenBuf1x Out-of-Bounds Write

**Commit**: `be3f979`

### Symptom
After the emulator starts (ROM loaded), in some situations it immediately crashes or shows memory corruption, with massive garbage or a crash on the very first frame.

### Root Cause
In `PPU.cs`, the `scanline` static field is initialized to `-1` (the NES PPU starts before pre-render on power-up):

```csharp
static public int ppu_cycles_x = 0, scanline = -1;
```

Inside the cycle-accurate PPU's `ppu_step_new()`, the background-off fill loop check condition was:

```csharp
// Before (wrong condition)
if (scanline < 240)
{
    if (ppu_cycles_x == 0 && !ShowBackGround)
    {
        int scanOff = scanline << 8;  // = (-1) << 8 = -256 !!
        for (int i = 0; i < 256; i++)
        {
            ScreenBuf1x[scanOff + i] = bgColor;  // writes ScreenBuf1x[-256+i], out-of-bounds!
            Buffer_BG_array[scanOff + i] = 0;
        }
    }
    ...
}
```

`-1 < 240` is true, and at power-up `ShowBackGround=false`, triggering the loop.
`(-1) << 8 = -256`, writing to `ScreenBuf1x[-256]` accesses unallocated memory, causing memory corruption.

### Affected Location
`PPU.cs`: outer condition guard for background fill loop and sprite render trigger in `ppu_step_new()`.

### Fix
```csharp
// After: add scanline >= 0 lower-bound check
if (scanline >= 0 && scanline < 240)
{
    if (ppu_cycles_x == 0 && !ShowBackGround)
    {
        int scanOff = scanline << 8;  // scanline guaranteed >= 0, safe
        ...
    }
    if (ppu_cycles_x == 257)
        RenderSpritesLine();
}
```

`scanline = -1` only exists for the instant the emulator has just started, before the first PPU tick. During normal gameplay the scanline cycles between 0–261 and never hits this boundary condition.

## 14. Cycle-Accurate PPU: MMC3 IRQ Fires at PPU Cycle 4 Instead of Cycle 260

**Commit**: `9375bd0`

### Symptom
After the cycle-accurate PPU rewrite, MMC3 games like SMB3 show severe corruption in the lower portion of the screen (status bar area, roughly scanlines 192–239): scroll split position off, CHR bank switch timing wrong, manifesting as completely garbled tile patterns in the lower half.

### Root Cause
**Original batch-render IRQ timing**: calls `Mapper04step_IRQ()` at `ppu_cycles_x == 260` per scanline (in the middle of the sprite fetch section), which corresponds to the NES hardware A12 rising-edge count point.

**Cycle-accurate version's incorrect implementation**: uses `Clock_A12()` to detect CHR address bit 12 rising edges, called at BG tile fetch **phase 4** (cycle 4 per scanline = the 4th cycle of the first tile):

```csharp
case 4:
    ioaddr = BgPatternTableAddr | (NTVal << 4) | ((vram_addr >> 12) & 7);
    Clock_A12(ioaddr);  // A12 goes 0→1 → immediately calls IRQInterrupt()!
    break;
```

**Timing difference impact on CPU**: `IRQInterrupt()` is a **synchronous call** that immediately modifies CPU PC and SP:
```csharp
public static void IRQInterrupt()
{
    // directly pushes PC, jumps to IRQ vector; CPU executes IRQ handler on next step
    r_PC = (ushort)(Mem_r(0xfffe) | (Mem_r(0xffff) << 8));
    Interrupt_cycle = 7;
}
```

| Scheme | IRQ fires at PPU cycle | Handler starts (+7 CPU × 3) | $2005/$2006 write timing |
|--------|------------------------|------------------------------|--------------------------|
| Original (cycle 260) | cycle 260 | cycle ≈ 281 | near scanline boundary |
| Wrong (cycle 4) | cycle 4 | cycle ≈ 25 | **mid-scanline** |

The IRQ fires **256 PPU cycles early** (≈ 85 CPU cycles), causing the game's IRQ handler to write to scroll registers (`$2005`/`$2006`) **mid-scanline** instead of at the scanline boundary. SMB3's split-scroll effect depends on writing at the correct time; mid-scanline writes cause lower-half horizontal scroll corruption.

**Additional issue**: `Clock_A12` at sprite start point (cycle 257):
```csharp
Clock_A12(Spritesize8x16 ? 0 : SpPatternTableAddr);
```
Passes `0` (A12=0) for 8x16 sprites, but in practice some 8x16 sprite tiles access the $1000 region (A12=1). This approximation causes unstable IRQ counting on different ROMs.

### Affected Location
`PPU.cs`:
- `Clock_A12()` function itself
- `Clock_A12()` calls in `ppu_rendering_tick()` cases 4 and 6
- `Clock_A12()` call at cycle 257 in `ppu_step_new()`

### Fix
Remove `Clock_A12()` and all its call sites, restoring the original **fixed cycle 260 trigger** scheme:

```csharp
// After (inside ppu_step_new)

// Visible scanlines (0–239)
if (scanline >= 0 && scanline < 240)
{
    ...
    // MMC3 IRQ: clock once per visible scanline at cycle 260
    if (ppu_cycles_x == 260 && renderingEnabled && mapper == 4)
        (MapperObj as Mapper004).Mapper04step_IRQ();
}

// Pre-render scanline 261 also clocks (consistent with original ppu_step, see Bug 9)
if (scanline == 261 && ppu_cycles_x == 260 && renderingEnabled && mapper == 4)
    (MapperObj as Mapper004).Mapper04step_IRQ();
```

**CHR tile read timing unaffected**: `MapperR_CHR()` calls for BG tiles still execute at the correct PPU cycles in phases 5/7, independent of the IRQ trigger timing. The cycle-accurate advantages are fully preserved.

### Why Not Continue Using A12 Detection?
The fundamental problem with A12 detection is that `IRQInterrupt()` is a synchronous call with no mechanism to "defer delivery to the scanline boundary." To correctly fire the IRQ near cycle 260, an IRQ scheduling mechanism is needed (pending IRQ processed at the next CPU instruction boundary). Under the current architecture, the simplest and equivalently correct approach is to return to the fixed cycle 260 trigger.
