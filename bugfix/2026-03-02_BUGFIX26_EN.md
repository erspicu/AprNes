# BUGFIX 26 – D-pad Direction Key Configuration Root Cause: Imprecise Axis Values

**Date**: 2026-03-02
**Version**: AprNes
**Category**: Bug Fix

---

## Problem

D-pad direction keys in DirectInput D mode cannot be captured at all in the configuration UI (even though diagnostic tool JoyTest confirms `event_type=0 AXIS` events are being generated).

---

## Diagnosis Process

Built `tools/JoyTest.cs` as a standalone diagnostic tool, confirming:
- D-pad D mode does produce `event_type=0 AXIS` events ✓
- But configuration UI fields show no reaction at all

Tracing found the true root cause.

---

## Root Cause

`JoyPadWayName()` requires axis values to **exactly equal 0 or 65535**:

```csharp
private string JoyPadWayName(string xy_name, int value)
{
    if (xy_name == "X")
    {
        if (value == 0) return "LEFT";
        if (value == 65535) return "RIGHT";
    }
    if (xy_name == "Y")
    {
        if (value == 0) return "UP";
        if (value == 65535) return "DOWN";
    }
    return "";  // ← any other value returns empty string
}
```

The original `AXIS_NOISE` filter only handled noise **near center** (±256 of 32767), but if D-pad extreme values (min/max) are off (e.g. `1`, `100`, `65434`, etc.), `JoyPadWayName` returns `""` causing:

1. **During configuration**: `Setup_JoyPad_define()` returns early because `name == ""`, field shows nothing, keymap not written
2. **During gameplay**: gaming lookup key includes empty string, never finds keymap → direction keys have no effect

Scenarios that can trigger this:
- `DIPROP_RANGE` SetProperty call fails (device does not support it, silently ignored), axis still uses hardware native range (not 0~65535)
- Some gamepad manufacturers' D-pads do not return exact extreme values

---

## Fix

Added `SnapAxisDigital()` in `joystick.cs`, snapping all axis values to three discrete values:

```csharp
static int SnapAxisDigital(int v)
{
    if (v < 16384) return 0;       // lower half → negative direction
    if (v > 49151) return 65535;   // upper half → positive direction
    return 32767;                   // middle → neutral
}
```

Applied to:
- **DirectInput** `lX`/`lY` (after AXIS_NOISE center filter, then snap)
- **XInput** analog stick X/Y (after deadzone filter, then snap)
- **POV hat → XY** already outputs precise values, no change needed

Before/after comparison:

| Gamepad reported value | Before fix | After fix |
|------------------------|-----------|----------|
| 0 | "UP" ✓ | "UP" ✓ |
| 1 | "" ✗ | "UP" ✓ |
| 32767 | neutral, skipped ✓ | neutral, skipped ✓ |
| 65534 | "" ✗ | "DOWN" ✓ |
| 65535 | "DOWN" ✓ | "DOWN" ✓ |
| -1000 (DIPROP_RANGE failed) | "" ✗ | "UP" ✓ |
