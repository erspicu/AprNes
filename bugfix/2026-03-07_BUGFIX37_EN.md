# BUGFIX37: Frame Counter IRQ deferred clear + unconditional flag set

**Date**: 2026-03-07
**Affected test**: AccuracyCoin P14 $0467 (Frame Counter IRQ) err=7 → PASS
**Baseline change**: AC 111/136 → 112/136 PASS, blargg 174/174 unchanged

## Problem Description

AccuracyCoin Frame Counter IRQ test has 24 sub-tests, including:
- Tests 6/7: SLO abs,X dummy read + real read of $4015; whether the IRQ flag is cleared before the second read depends on APU get/put cycle parity
- Tests I-L: When apuintflag=true, frame counter IRQ flag should still be temporarily set for 2 cycles

## Root Cause Analysis

1. **$4015 read IRQ clear timing**: The original code immediately set `statusframeint = false` in `apu_r_4015()`. On real NES hardware, the clear is deferred until the next APU "get" cycle.

2. **apuintflag suppress behavior**: The original code gated `statusframeint` assignment with `!apuintflag` in `clockframecounter()`. Real hardware unconditionally sets the IRQ flag for 2 cycles even when `apuintflag=true`; suppression only occurs on the 3rd cycle.

3. **IRQ line driving**: The original code directly used `statusframeint` to drive `irqLineCurrent`. The `!apuintflag` check needs to be added here.

## Fix Details

### APU.cs

1. **Added `frameIrqClearPending` flag** — $4015 read sets this flag instead of immediately clearing
2. **Deferred clear handled in `apu_step()`** — Clear is only executed on an APU "get" cycle where `(cpuCycleCount & 1) == 0`, placed before the frame counter assertion
3. **`irqAssertCycles` processing removes `!apuintflag` gate** — flag is unconditionally set; if `apuintflag=true` on the last cycle, it is then cleared
4. **`clockframecounter()` removes `!apuintflag` gate** — IRQ flag is unconditionally set

### MEM.cs

5. **`irqLineCurrent` adds `!apuintflag`** — `(statusframeint && !apuintflag) || statusdmcint || statusmapperint`

- blargg 174/174 PASS (no regression)
- AccuracyCoin 112/136 PASS (+1: Frame Counter IRQ)
