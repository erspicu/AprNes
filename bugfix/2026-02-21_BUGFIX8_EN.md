## 33. MMC3 IRQ A12 Clocking: Changed from Hardcoded Cycle 260 to PPU Address Bus A12 Rising Edge Driven

### Symptom

16 of 174 test ROMs FAIL for MMC3 (only 1 of 17 MMC3 tests passing). Main error messages:

- "Should decrement when A12 is toggled via $2006"
- All "A12 clocking" tests fail
- "MMC3 rev B" behavior incorrect

### Root Cause

The MMC3 (Mapper 004) IRQ counter should be driven by **A12 (bit 12) rising edges on the PPU address bus**. On real hardware, the following events all produce A12 transitions on the PPU address bus:

1. **BG tile fetch** (cycles 0-255, 320-335): NT/AT fetch → A12=0; pattern fetch → A12=BgPatternTableAddr bit 12
2. **Sprite tile fetch** (cycles 256-319): dummy NT fetch → A12=0; sprite pattern fetch → A12=SpPatternTableAddr bit 12
3. **$2006 (PPUADDR) write**: CPU sets vram_addr, A12 changes accordingly
4. **$2007 (PPUDATA) read/write**: vram_addr increments, A12 may change
5. **Pre-render scanline** (261) vert copy: vram_addr overwritten by vram_addr_internal

The original implementation **only hardcoded a single call** to `Mapper04step_IRQ()` at cycle 260, tracking no A12 state at all. As a result, $2006/$2007 manual A12 changes cannot clock the IRQ counter, and scanline timing is also incorrect.

### Fix

#### 1. Mapper004.cs — Add A12 Tracking and Low-Pass Filter

```csharp
// New fields
int lastA12 = 0;
int a12LowSince = -100;  // PPU absolute cycle when A12 last went low
const int A12_FILTER = 16;

// New method
public void NotifyA12(int address, int ppuAbsCycle)
{
    int a12 = (address >> 12) & 1;
    if (a12 != 0 && lastA12 == 0)
    {
        // Rising edge 0→1: clock IRQ counter if A12 was low long enough
        int elapsed = ppuAbsCycle - a12LowSince;
        if (elapsed < 0) elapsed += 341 * 262;
        if (elapsed >= A12_FILTER)
            Mapper04step_IRQ();
    }
    else if (a12 == 0 && lastA12 != 0)
    {
        // Falling edge 1→0: record when A12 went low
        a12LowSince = ppuAbsCycle;
    }
    lastA12 = a12;
}
```

The low-pass filter purpose: real MMC3 hardware has a built-in filter that ignores A12 pulses shorter than 16 PPU cycles. This avoids spurious clocks from rapid NT/pattern switching during BG tile fetches.

#### 2. PPU.cs — Add NotifyMapperA12() Helper

```csharp
[MethodImpl(MethodImplOptions.AggressiveInlining)]
static void NotifyMapperA12(int address)
{
    ((Mapper004)MapperObj).NotifyA12(address, scanline * 341 + ppu_cycles_x);
}
```

#### 3. PPU.cs — Notify A12 on BG Tile Fetch

In `ppu_rendering_tick()` cases 0/2/4/6, after ioaddr is set, add one line each:
```csharp
if (mapper == 4) NotifyMapperA12(ioaddr);
```

Cases 0/2 are NT/AT fetches (address 0x2000+, A12=0);
cases 4/6 are pattern fetches (A12 determined by BgPatternTableAddr).

#### 4. PPU.cs — Simulate A12 for Sprite Fetches

Cycles 260-319, every 8 cycles simulate dummy NT fetch + sprite pattern fetch:
```csharp
else if (ppu_cycles_x >= 260 && ppu_cycles_x < 320 && mapper == 4)
{
    int phase = (ppu_cycles_x - 256) & 7;
    if (phase == 0) NotifyMapperA12(0x2000);                // dummy NT, A12=0
    else if (phase == 4) NotifyMapperA12(SpPatternTableAddr); // sprite CHR fetch
}
```

#### 5. PPU.cs — Remove Hardcoded Cycle 260

Remove two hardcoded calls:
- `Mapper04step_IRQ()` at cycle 260 for visible scanlines (0-239)
- `Mapper04step_IRQ()` at cycle 260 for pre-render scanline (261)

#### 6. PPU.cs — $2006 Second Write Notification

After `vram_addr = vram_addr_internal;` in `ppu_w_2006()`:
```csharp
if (mapper == 4) NotifyMapperA12(vram_addr);
```

#### 7. PPU.cs — Pre-render Vert Copy Notification

After vram_addr copy at scanline 261 / cycles 280-304:
```csharp
if (mapper == 4) NotifyMapperA12(vram_addr);
```

#### 8. MEM.cs — Notify After $2007 Read/Write

In all `ppu_read_fun` (5 locations) and `ppu_write_fun` (3 locations) lambdas, after each `vram_addr` increment, add:
```csharp
if (mapper == 4) NotifyMapperA12(vram_addr);
```
