# BUGFIX37: Frame Counter IRQ deferred clear + unconditional flag set

**日期**: 2026-03-07
**影響測試**: AccuracyCoin P14 $0467 (Frame Counter IRQ) err=7 → PASS
**基線變化**: AC 111/136 → 112/136 PASS, blargg 174/174 不變

## 問題描述

AccuracyCoin Frame Counter IRQ 測試有 24 個子測試，其中：
- Tests 6/7: SLO abs,X 對 $4015 的 dummy read + real read，依據 APU get/put cycle parity 決定 IRQ flag 是否在第二次 read 前被 clear
- Tests I-L: apuintflag=true 時 frame counter IRQ flag 仍應暫時 set 2 cycles

## 根因分析

1. **$4015 read IRQ clear timing**: 原始碼在 `apu_r_4015()` 中立即 `statusframeint = false`。真實 NES 硬體的 clear 會延遲到下一個 APU "get" cycle 才生效。

2. **apuintflag suppress 行為**: 原始碼在 `clockframecounter()` 中用 `!apuintflag` 門控 `statusframeint` 設定。真實硬體即使 `apuintflag=true`，IRQ flag 仍會被無條件 set 2 cycles，只在第 3 cycle 被 suppress。

3. **IRQ line 驅動**: 原始碼直接用 `statusframeint` 驅動 `irqLineCurrent`。需要加上 `!apuintflag` 判斷。

## 修復內容

### APU.cs

1. **新增 `frameIrqClearPending` flag** — $4015 read 設定此 flag，不再立即 clear
2. **`apu_step()` 中處理 deferred clear** — 在 `(cpuCycleCount & 1) == 0` 的 APU "get" cycle 才執行 clear，且位於 frame counter assertion 之前
3. **`irqAssertCycles` 處理移除 `!apuintflag` 門控** — flag 無條件 set，最後 cycle 若 `apuintflag=true` 則 clear
4. **`clockframecounter()` 移除 `!apuintflag` 門控** — IRQ flag 無條件 set

### MEM.cs

5. **`irqLineCurrent` 加上 `!apuintflag`** — `(statusframeint && !apuintflag) || statusdmcint || statusmapperint`

## 測試結果

- blargg 174/174 PASS (無回歸)
- AccuracyCoin 112/136 PASS (+1: Frame Counter IRQ)
