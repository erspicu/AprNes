## Fix 1: Stale BG Shift Registers (P19, $0483, PASS)

### Problem
When rendering is disabled mid-scanline and re-enabled on the next scanline, the PPU's
BG shift registers retain stale tile data. Sprite 0 hit detection should use this stale
data to detect hits immediately when rendering resumes. Our emulator failed because:

1. `PrecomputeSprite0Line()` runs at dot 0 of each scanline
2. If rendering is disabled at dot 0, the function early-returns (`!ShowBG && !ShowSpr`)
3. `sprite0_on_line = false` -> no hit detection possible for the entire scanline
4. When rendering re-enables mid-scanline, sprite 0 was never set up

### Root Cause
On real hardware, when rendering is disabled, sprite counters and shift registers are
frozen. When re-enabled, sprites start outputting immediately (all sprites treated as
X=0 relative to the current dot if counters weren't loaded at dot 339).

### Fix (PPU.cs: ppu_w_2001)
When transitioning disabled -> enabled during a visible scanline:
- Re-run `PrecomputeSprite0Line()` (now rendering is enabled, the guard passes)
- Set `sprite0_line_x = ppu_cycles_x` (frozen sprite counter starts at current dot)

```csharp
if (newRenderingEnabled && !sprite0_on_line && !isSprite0hit && ppu_cycles_x > 0)
{
    PrecomputeSprite0Line();
    if (sprite0_on_line)
        sprite0_line_x = ppu_cycles_x;
}
```

This passes all 4 sub-tests:
- Test 1: Sprite zero hit verification (prerequisite)
- Test 2: False positive check (specific tile pattern doesn't overlap)
- Test 3: Stale BG data causes hit (tile overlaps with frozen sprite)
- Test 4: Sprite at X=$80 still hits at current dot (dot 339 was missed)

## Fix 2: Deferred Load DMA Model (architectural improvement, no AC change)

### Change
Load DMA ($4015 write) now uses a deferred model matching Mesen2's `ProcessPendingDma`:

1. When `dmcLoadDmaCountdown` reaches 0 in `apu_step()`, sets `dmcDmaPending = true`
   instead of calling `dmcfillbuffer()` immediately
2. `dmcDmaPending` is checked at the start of `Mem_r()` and `ZP_r()`, BEFORE `tick_pre()`
3. This matches Mesen2 where `ProcessPendingDma` fires before `StartCpuCycle`

### Rationale
The deferred model ensures DMA phantom reads use the correct CPU bus address (the
address the CPU is about to read, not the address from the previous cycle). This is
necessary for tests like DMA+$2002R and DMA+$4015R where phantom reads must trigger
register side effects.

### Current Status
The model is architecturally correct but DMA+$2002R and DMA+$4015R still fail (err=2)
due to a 1-cycle cumulative timing drift. The test's clockslide + SLO $4015 sync
detects the wrong cycle parity because our frame counter IRQ timing has a sub-cycle
offset. The countdown formula produces delay=2 (DMA fires 1 cycle too early) instead
of delay=3. Fixing this requires sub-cycle M2 duty cycle modeling.

### Files Modified
- `APU.cs`: `apu_step()` sets `dmcDmaPending` instead of calling `dmcfillbuffer()`;
  `apu_4015()` countdown formula unchanged
- `MEM.cs`: `Mem_r()` and `ZP_r()` check `dmcDmaPending` before `tick_pre()`
