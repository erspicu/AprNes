# BUGFIX 27 – GitHub Pages Deployment Triple-Chained Issues

**Date**: 2026-03-03
**Version**: AprNesWasm
**Category**: Deployment Fix

---

## Issue Overview

After AprNesWasm's first deployment to GitHub Pages, three consecutive issues prevented the website from working correctly.

---

## Issue 1: Massive 404s (CSS / JS / WASM Files Not Found)

### Symptom
Browser Network tab shows large numbers of 404s under `_framework/`, `lib/`, `css/` and other folders.

### Root Cause
`dotnet publish` generates Brotli-compressed (`.br`) and gzip-compressed (`.gz`) files by default.
Blazor's bootloader prefers requesting `.br` compressed versions, but GitHub Pages does not serve `Content-Encoding: br` headers, so the browser cannot decompress them and loading fails.

### Fix
Add one line to `AprNesWasm.csproj` to disable compression:

```xml
<BlazorEnableCompression>false</BlazorEnableCompression>
```

`dotnet publish` no longer generates `.br` / `.gz` files; GitHub Pages serves the original files directly.

---

## Issue 2: CSS / JS Path Errors (base href Not Replaced)

### Symptom
All resources make requests to `https://erspicu.github.io/lib/...`, `https://erspicu.github.io/_framework/...` (missing the `/AprNes/` sub-path), all returning 404.

### Root Cause
`deploy_wasm.bat` uses `^` line continuation to chain PowerShell commands:

```bat
powershell -NoProfile -Command ^
  "(Get-Content ...) -replace ..."
```

With cmd's `^` line continuation, leading spaces on the next line are included in the command, causing PowerShell command parsing to fail. The `base href` in `index.html` stays at `"/"` instead of being patched to `/AprNes/`.

### Fix
Changed to a single-line command without continuation:

```bat
powershell -NoProfile -Command "(Get-Content 'deploy_tmp\index.html') -replace '<base href=\"/\" />', '<base href=\"/AprNes/\" />' | Set-Content 'deploy_tmp\index.html'"
```

## Issue 3: Subresource Integrity (SRI) Hash Mismatch

### Symptom
Chrome Console error:
```
Failed to find a valid digest in the 'integrity' attribute for resource
'https://erspicu.github.io/AprNes/_framework/dotnet.8thwiicode.js'
with computed SHA-256 integrity 'sY6v9Jkaxjb7...'.
The resource has been blocked.
```

### Root Cause
Windows git defaults to `core.autocrlf=true`, converting LF line endings in JS files to CRLF on push.
This changes the file content, causing the SHA-256 hash computed by the browser to differ from the `integrity` attribute embedded in `index.html`, triggering SRI security blocking.

The large number of git warnings during deployment are symptoms of this issue:
```
warning: in the working copy of '_framework/blazor.webassembly.xxx.js',
LF will be replaced by CRLF the next time Git touches it
```

### Fix
Set `core.autocrlf=false` in the temporary git repo used for deployment:

```bat
git init -b gh-pages -q
git config core.autocrlf false   ← add this line
git add -A
```

## Recommended Troubleshooting Order

For future Blazor WASM deployments to GitHub Pages, check in order:

1. **Completely blank website** → Confirm `.nojekyll` exists and GitHub Pages Settings is configured for `gh-pages` branch
2. **JS/WASM 404** → Confirm `<BlazorEnableCompression>false</BlazorEnableCompression>`
3. **CSS path errors** → Confirm `index.html` `<base href>` has been patched to sub-path
4. **SRI errors** → Confirm deploy repo has `core.autocrlf false`
