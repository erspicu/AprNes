# BUGFIX17: Sprite timing — per-pixel hit detection + cycle-accurate overflow + hardware bug

**Date**: 2026-02-22
**Baseline**: 161 PASS / 13 FAIL → **165 PASS / 9 FAIL** (+4 PASS)

## Problems

- `sprite_hit_tests/09.timing_basics` — FAILED #3: upper-left corner sprite 0 hit too late
- `sprite_hit_tests/10.timing_order` — FAILED #3: upper-left corner sprite 0 hit too late
- `sprite_overflow_tests/3.Timing` — FAILED #5: overflow flag set too late for first scanline
- `sprite_overflow_tests/4.Obscure` — FAILED #2: sprite overflow hardware bug (byte offset) not implemented

## Root Causes

### 1. Sprite 0 hit delay (tests 09, 10)

`RenderBGTile()` renders 8 pixels in batch at phase 7 (cycle 7, 15, 23...) and checks sprite 0 hit. If the hit is on pixel 0, the flag is not set until cycle 7. Real NES checks every dot, as early as dot 2. Delay of up to 7 PPU dots.

### 2. Sprite overflow set at cycle 257 (test 3)

`RenderSpritesLine()` scans all 64 sprites at cycle 257 and immediately sets overflow. On real NES, sprite evaluation happens during dots 65-256: in-range sprites take 8 cycles, out-of-range sprites take 2 cycles. 9 sprites at Y=0 should set overflow at ~cycle 129.

### 3. Overflow bug not implemented (test 4)

After finding 8 sprites, the real NES PPU has a bug: byte offset m increments with sprite index n (mod 4), causing tile/attr/X bytes to be read as Y coordinates. We were always reading from byte 0 (Y).

## Fixes (PPU.cs)

### Fix 1: Per-pixel sprite 0 hit detection

- Removed sprite 0 hit check from `RenderBGTile()`
- Added per-dot detection (dot 2-255) **before** `ppu_rendering_tick()` in `ppu_step_new()`
- Uses shift register holding the previous phase 7 loaded value to compute current dot's BG pixel

### Fix 2: Cycle-accurate sprite overflow

- Added `PrecomputeOverflow()` function, called at cycle 0 of each visible scanline
- Simulates NES sprite evaluation timing: starts at dot 65, in-range 8 cycles, out-of-range 2 cycles
- Records the cycle at which overflow should be set; checked every cycle in `ppu_step_new()`

### Fix 3: Sprite overflow hardware bug

- In `PrecomputeOverflow()`, after finding 8 sprites:
  - 9th sprite reads byte 0 (Y) normally
  - If not in range: n++ and m = (m+1) & 3
  - Subsequent reads use byte m (cycles through 1→2→3→0→1...)
- Removed old overflow detection logic from `RenderSpritesLine()`
