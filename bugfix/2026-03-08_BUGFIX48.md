# BUGFIX48: AccuracyCoin P19 $2004 Stress Test

**日期**: 2026-03-08
**基線**: 120/136 → **121/136** (+1)
**修復測試**: P19 Test 6 ($2004 Stress Test) err=1 → PASS

## 根因

P19 $2004 Stress Test 在 scanline 128 的每個 dot (0-340) 讀取 $2004，跨越 21 個 frame 收集完整資料，
然後驗證每個 dot 的返回值是否符合 PPU sprite evaluation 狀態機的行為。

四個問題：

### 1. ppu_r_2004 範圍邊界未補償 tick-before-read 時序偏移

我們的 tick() 模型在 bus read 前先處理 3 個 PPU dots。$2004 讀取時 cx=N+1 對應硬體 dot N。
ppu_r_2004 的範圍檢查需要 +1 偏移來補償。

### 2. Clear phase 返回 oamCopyBuffer 而非硬編碼 $FF

Dots 1-64 secondary OAM 清除階段，$2004 應始終返回 $FF，與 oamCopyBuffer 無關。

### 3. Attribute byte masking 位置錯誤

硬體在從 primary OAM 讀取 attribute byte 時就在內部匯流排上 mask bits 2-4。
之前在寫入 secondary OAM 時才 mask，導致 $2004 讀取返回未 mask 的原始值。
測試期望 odd/even dot 都返回 masked 值（如 $61 而非 $7D）。

### 4. Post-evaluation phase 缺少 secondary OAM 讀取和地址推進

所有 64 個 sprite 檢查完畢後（oamCopyDone），even dots 應從 secondaryOAM[secOAMAddr]
讀取值到 oamCopyBuffer（寫入變為讀取），且 spriteEvalAddrH 應繼續推進以循環 primary OAM。

## 修復

### PPU.cs — ppu_r_2004()

範圍邊界 +1 補償：
- Clear phase: cx [2, 65]（原 [1, 64]），返回硬編碼 $FF
- Eval phase: cx [66, 257]（原 [65, 256]）
- Sprite tile fetch: cx [258, 321]（原 [257, 320]）

### PPU.cs — SpriteEvalTick()

Odd dot 讀取 primary OAM 時，若 spriteEvalAddrL == 2（attribute byte），
立即 mask: `oamCopyBuffer &= 0xE3`

### PPU.cs — SpriteEvalWrite()

Post-eval phase（oamCopyDone && secOAMAddr < 0x20）：
- 從 secondaryOAM[secOAMAddr] 讀取到 oamCopyBuffer
- 推進 spriteEvalAddrH（繼續循環 primary OAM addresses）
