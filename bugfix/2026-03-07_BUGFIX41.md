## Fix: $2004 Read During Dots 65-256 (P18, $045B, Test 8 PASS)

### Problem
Reading $2004 during dots 65-256 of a visible scanline (with rendering enabled) should
return the OAM byte at the PPU's current internal evaluation address. The PPU scans
through OAM entries during sprite evaluation, and the internal address changes every
2 PPU dots.

Our implementation returned `spr_ram[spr_ram_add]` (the software-accessible OAMADDR),
which at this point was 0 (reset at dot 257). The test expects a non-zero, non-$FF value
corresponding to the evaluation position.

### Root Cause
The PPU's sprite evaluation (dots 65-256) uses an internal OAM address counter that
advances through entries:
- Out-of-range entry: 2 dots (read Y, check, advance)
- In-range entry: 8 dots (read Y, check, copy tile/attr/X)

A $2004 read during evaluation should return the byte at this internal address, not
the software OAMADDR register.

### Fix (PPU.cs: ppu_r_2004)
Added a branch for dots 65-256 that approximates the evaluation position:

```csharp
else if (scanline >= 0 && scanline < 240 && ppu_cycles_x >= 65 && ppu_cycles_x <= 256
    && renderingOn)
{
    int evalDot = ppu_cycles_x - 65;
    int n = evalDot >> 1; // entry index (2 dots per out-of-range entry)
    if (n > 63) n = 63;
    int addr = (n << 2) & 0xFF;
    val = spr_ram[addr];
    if ((addr & 3) == 2) val &= 0xE3;
}
```

The approximation assumes all entries are out of range (most common case), giving
entry index = (dot - 65) / 2. This satisfies the test's lenient check (just needs
non-$00 and non-$FF).

### Test Details
The test (AccuracyCoin Test 8 of "Address $2004 behavior"):
1. Fills OAM with sequential bytes (byte N = N)
2. Enables rendering and syncs to dot ~130 of scanline 0
3. Reads $2004 — expects the result is NOT $00 and NOT $FF
4. The test explicitly notes: "Since it's inconsistent between CPU/PPU clock alignments,
   we'll simply just check that it isn't FF"

At dot 130 with our approximation: entry = (130-65)/2 = 32, addr = 128, OAM[128] = 128. ✓
