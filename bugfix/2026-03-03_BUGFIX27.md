# BUGFIX 27 – GitHub Pages 部署三連環問題

**日期**：2026-03-03  
**版本**：AprNesWasm  
**分類**：Deployment Fix  

---

## 問題概述

AprNesWasm 首次部署到 GitHub Pages 後，連續出現三個問題導致網站無法正常運作。

---

## 問題一：大量 404（CSS / JS / WASM 檔案找不到）

### 現象
瀏覽器 Network tab 顯示 `_framework/`、`lib/`、`css/` 等資料夾下大量 404。

### 根本原因
`dotnet publish` 預設產生 Brotli 壓縮檔（`.br`）和 gzip 壓縮檔（`.gz`）。  
Blazor 的 bootloader 會優先請求 `.br` 壓縮版本，但 GitHub Pages 沒有設定 `Content-Encoding: br` header，瀏覽器無法解壓，導致載入失敗。

### 修復
在 `AprNesWasm.csproj` 加入一行停用壓縮：

```xml
<BlazorEnableCompression>false</BlazorEnableCompression>
```

`dotnet publish` 不再產生 `.br` / `.gz`，GitHub Pages 直接 serve 原始檔案。

---

## 問題二：CSS / JS 路徑錯誤（base href 未替換）

### 現象
所有資源都向 `https://erspicu.github.io/lib/...`、`https://erspicu.github.io/_framework/...` 發請求（缺少 `/AprNes/` 子路徑），全部 404。

### 根本原因
`deploy_wasm.bat` 使用 `^` 換行符號串接 PowerShell 命令：

```bat
powershell -NoProfile -Command ^
  "(Get-Content ...) -replace ..."
```

cmd 的 `^` 換行後，下一行開頭的空格會被帶入命令，導致 PowerShell 命令解析失敗，`index.html` 的 `base href` 始終維持 `"/"` 而非 `/AprNes/`。

### 修復
改為不換行的單行命令：

```bat
powershell -NoProfile -Command "(Get-Content 'deploy_tmp\index.html') -replace '<base href=\"/\" />', '<base href=\"/AprNes/\" />' | Set-Content 'deploy_tmp\index.html'"
```

## 問題三：Subresource Integrity (SRI) hash 不符

### 現象
Chrome Console 錯誤：
```
Failed to find a valid digest in the 'integrity' attribute for resource
'https://erspicu.github.io/AprNes/_framework/dotnet.8thwiicode.js'
with computed SHA-256 integrity 'sY6v9Jkaxjb7...'.
The resource has been blocked.
```

### 根本原因
Windows git 預設 `core.autocrlf=true`，在 push 時會將 JS 檔案的 LF 換行轉成 CRLF。  
這改變了檔案內容，使得瀏覽器計算出的 SHA-256 hash 與 `index.html` 內嵌的 `integrity` 屬性不符，觸發 SRI 安全封鎖。

deploy 時 git 輸出大量警告即為此問題的徵兆：
```
warning: in the working copy of '_framework/blazor.webassembly.xxx.js',
LF will be replaced by CRLF the next time Git touches it
```

### 修復
在 deploy 用的暫存 git repo 設定 `core.autocrlf=false`：

```bat
git init -b gh-pages -q
git config core.autocrlf false   ← 新增此行
git add -A
```

## 問題排查順序建議

未來 GitHub Pages 部署 Blazor WASM 若出現問題，依序檢查：

1. **網站完全空白** → 確認 `.nojekyll` 存在，GitHub Pages Settings 已設定 `gh-pages` 分支
2. **JS/WASM 404** → 確認 `<BlazorEnableCompression>false</BlazorEnableCompression>`
3. **CSS 路徑錯誤** → 確認 `index.html` 的 `<base href>` 已 patch 為子路徑
4. **SRI 錯誤** → 確認 deploy repo 設定 `core.autocrlf false`
